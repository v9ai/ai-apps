Chapter One: Why Eval Matters

Welcome to this deep dive on the evaluation harness of the BS Detector, a five-agent pipeline that verifies legal briefs for factual and citation accuracy. Over the next hour or so, we are going to walk through every layer of this evaluation system, from the philosophy behind it to the exact matching algorithms, from the ground truth design to the persistence layer. By the end, you will understand not just what the eval harness does, but why every design decision was made the way it was.

Let us start with why evaluation matters so much for legal AI.

When you build an AI system that helps judges assess legal briefs, the cost of error is fundamentally different from most software. A false positive, where the system wrongly accuses an attorney of citing a fabricated case, wastes court time and erodes judicial trust. A false negative, where the system misses a real error in a brief, means a flawed legal argument goes unchallenged. These errors are not symmetrical. In most AI applications, false positives and false negatives are roughly equally bad. In legal AI, the asymmetry is extreme.

Consider the real-world consequences. After the Mata v. Avianca case in twenty twenty-three, where attorneys submitted entirely fabricated case citations generated by ChatGPT, judges now assume AI output is unverified until proven otherwise. The attorneys were fined five thousand dollars. In the Crabill case in Colorado, an attorney received a one-year suspension, the first AI-related attorney suspension in the United States. The escalation pattern is clear. Courts are becoming less tolerant of AI errors, not more tolerant.

This asymmetry has a direct implication for evaluation strategy. You cannot just measure accuracy as a single number. You need to separately measure recall, which is how many real errors the system catches, and precision, which is how often the system avoids false accusations. And you need to measure hallucination, which is whether the system fabricates findings that have no basis in the source documents.

Here is the counterintuitive insight that drives the entire eval harness design: you build the evaluation before you tune the prompts. This is the opposite of how most people approach LLM development. The typical workflow is: write a prompt, look at the output, tweak the prompt, look again. That is prompt engineering by vibes. It works for chatbots and content generation. It fails catastrophically for legal AI because you cannot see precision and recall by looking at a single output. You need structured measurement against known ground truth.

The BS Detector eval harness was built before any prompt tuning happened. The eight planted errors in the test case were defined first. The matching algorithm was defined first. The six evaluation categories were defined first. Only then were the prompts written and refined, with each change measured against the established baseline. This is the eval-driven development philosophy: measurement first, optimization second.

One more foundational point. The eval harness itself must be more predictable than the system it evaluates. If you use an LLM to evaluate an LLM, you have introduced LLM uncertainty into the measurement of LLM accuracy. That is epistemologically problematic. It is like measuring a ruler with another ruler that might be warped. This is why the default evaluation in the BS Detector uses keyword matching, which is deterministic and transparent. The LLM-as-judge is available as an opt-in secondary signal, not as the primary measurement. We will explore this tension in depth later.


Chapter Two: The Four-Layer Architecture

The BS Detector evaluation system is not a single script. It is a four-layer architecture, each layer testing different things at different levels of abstraction.

The first layer is the Python eval harness. This is the main orchestrator in run evals dot py, a seven hundred sixteen line Python script that runs the full pipeline against test cases and measures output quality across six categories. It invokes the actual five-agent pipeline, captures the report, and runs thirty-one structured assertions against the output. This layer tests the integrated system: all five agents working together, end to end.

The second layer is the promptfoo pipeline-level evaluation defined in promptfoo config dot yaml. This is a two hundred fourteen line YAML file that defines twenty assertions using JavaScript, tested against the same pipeline output. Promptfoo is an open-source evaluation framework that provides a web viewer, caching, and parallel test execution. The Python harness and the promptfoo config share a cache, so the pipeline only runs once and both evaluation layers read the same report.

The third layer is agent-level evaluation. These are isolated promptfoo configs that test individual agents in isolation, separated from the rest of the pipeline. There are configs for citation extraction, fact checking, citation verification, and each one sends specific inputs directly to the agent's prompt and checks the output against ground truth. This layer lets you diagnose whether a failure is caused by a specific agent or by the agent interactions.

The fourth layer is prompt precision A/B tests. These configs run the same test cases against two versions of a prompt: a precise version with detailed instructions, schema definitions, and domain knowledge, and an imprecise version that is vague and underspecified. By measuring the difference in scores between precise and imprecise prompts, you quantify the measurable impact of prompt engineering choices.

There is also a reflection honesty evaluation that uses DeepSeek R-one as a judge to evaluate the project's own reflection document across six dimensions. We will cover that in its own chapter.

Why four layers instead of one? Because different layers catch different failures. The integrated Python harness catches pipeline-level issues: agents not passing data correctly, the orchestrator dropping results, or the synthesizer misranking findings. The agent-level evals catch prompt-level issues: the fact checker missing a date discrepancy, or the citation verifier hallucinating a holding. The A/B tests isolate the impact of prompt precision from other variables. And the promptfoo layer provides a standardized web viewer and reporting format that makes results shareable with non-engineers.

A single monolithic eval script would conflate all these concerns. When a test fails, you would not know whether the problem is in the prompt, the agent logic, the pipeline plumbing, or the evaluation itself. The four-layer architecture gives you diagnostic precision.


Chapter Three: Ground Truth Design

The ground truth is the foundation of any evaluation system. If the ground truth is wrong, every metric built on top of it is meaningless. Let us walk through how the BS Detector ground truth was designed.

The test case is Rivera v. Harmon Construction Group, a fictional motion for summary judgment in a California workplace injury case. The motion was written with eight specific errors planted into it. These are not random errors. Each one was chosen to test a different detection capability.

Error D-01 is a date discrepancy. The motion says the workplace incident occurred on March fourteenth, twenty twenty-one. Every other document, the police report, the medical records, and the witness statement, says March twelfth. This tests cross-document date consistency checking, one of the most basic verification tasks. It carries a weight of two because getting the date wrong affects the statute of limitations calculation, which has cascading legal consequences.

Error D-02 is about personal protective equipment. The motion claims Rivera was not wearing required PPE. But the police report confirms he was wearing a hard hat and harness, and the witness statement adds a high-visibility vest. This tests the system's ability to detect a direct factual contradiction between the motion and its supporting documents. It also carries a weight of two because falsely claiming a worker lacked safety equipment is a particularly egregious misrepresentation that shifts blame to the injured party.

Error D-03 is the Privette misquotation. The motion quotes Privette v. Superior Court as saying, quote, "a hirer is never liable for injuries sustained by an independent contractor's employees." The word "never" was inserted by the brief's author. The actual holding creates a rebuttable presumption, not absolute immunity. This is the most subtle error because it requires legal knowledge to detect. It tests citation accuracy at the holding level. Weight: two.

Error D-04 is a statute of limitations calculation error. The motion says Rivera filed "one year and three hundred sixty-two days after the incident," but this calculation uses the wrong March fourteenth date. With the correct March twelfth date, the elapsed time is different. This tests whether the system connects the date error to its downstream legal consequences. Weight: one.

Error D-05 is about retained control. The motion implies Harmon had no control over the work, which is central to the Privette defense. But the police report shows that Harmon's foreman Ray Donner directed Rivera's crew and the witness statement shows Donner dismissed safety concerns about the scaffolding. This tests cross-document inference, connecting statements from different sources to contradict a legal claim. Weight: two.

Error D-06 is a jurisdiction issue. The motion cites Dixon v. Lone Star, which is a Texas case, and Okafor v. Brightline, which is a Florida case. Neither is binding authority in California state court. This tests the system's ability to recognize out-of-state citations and flag them as non-binding. Weight: one.

Error D-07 is a scaffolding condition omission. The motion says nothing about the scaffolding's condition, but both the police report and witness statement document rust, bent coupling pins, and an unstable plywood base. This tests whether the system can detect a significant omission, not just a direct contradiction but the strategic absence of material information. Weight: one.

Error D-08 is about post-incident spoliation. The witness statement reveals that the scaffolding was rebuilt from scratch with new components after the incident, replacing the plywood base with proper concrete footings. This reconstruction destroyed physical evidence. The motion does not mention it. This tests whether the system can recognize potential destruction of evidence. Weight: one.

The weight assignments reflect legal impact. D-01, D-02, D-03, and D-05 get double weight because they involve either critical factual falsehoods or misrepresentation of binding legal standards. D-04, D-06, D-07, and D-08 get single weight because while significant, they involve downstream calculations, non-binding authority, omissions, or post-incident conduct rather than direct misrepresentation of core facts or law.

The total weight across all eight errors is twelve. The recall score is the sum of weights for detected errors divided by twelve. This means catching all four critical errors but missing all four standard errors gives you a weighted recall of eight out of twelve, or sixty-seven percent. Catching all four standard errors but missing all critical ones gives you four out of twelve, or thirty-three percent. The weighting ensures that the metric reflects legal significance, not just detection count.


Chapter Four: The Matching Algorithm

Now let us get into the mechanics of how the eval harness determines whether the pipeline has detected a planted error. This is the matching algorithm, and it has more nuance than you might expect.

The matching operates on extracted text from the pipeline report. The first step is text extraction. The function called extract searchable text takes the pipeline report and a list of sections to search. Each ground truth item specifies which sections to search in, either top findings, verified facts, verified citations, or a combination. For top findings, it concatenates the description, type, evidence items, and recommendation into a single lowercase text blob. For verified facts, it joins the summary, fact text, contradictory sources, and status. For verified citations, it joins the notes, discrepancies, citation text, claimed proposition, and status.

This extraction step matters because the pipeline output is deeply nested. A finding about the date error might appear in the description field of a top finding, or it might be in the summary field of a verified fact marked contradictory, or it might be split across the fact text and the contradictory sources list. By extracting and concatenating all relevant fields into flat text blobs, the matching algorithm can search across all of them without needing to know the exact nesting structure.

Once the text is extracted, the matching uses one of two modes defined by the match mode field in each ground truth item.

The first mode is called "any." In this mode, the check passes if any single keyword from the keywords list appears anywhere in the concatenated text. This mode is used for errors where a single keyword is specific enough to be a reliable indicator. For example, D-01 uses any mode with keywords "march twelve," "march fourteen," "date discrepancy," and "date inconsisten." The keyword "march twelve" alone is specific enough to indicate the system detected the date issue. You do not need additional confirmation.

The second mode is called "keyword plus signal." In this mode, the check requires both a keyword hit from the keywords list and a signal hit from the require also list. This mode is used for errors where a keyword alone is ambiguous. For example, D-02 uses keyword plus signal. The keywords are "harness," "ppe," "hard hat," and "protective equipment." But mentioning PPE alone does not mean the system detected a contradiction. The system might just be describing what the motion claims. So the require also list adds signal words: "contradict," "wearing," "false," "inconsisten," and "not supported." The check passes only if the text contains both a PPE keyword and a signal word indicating the system recognized the contradiction.

This dual-mode approach is a deliberate precision-recall tradeoff at the evaluation level. The any mode maximizes sensitivity: if the system mentions the date at all, it gets credit. The keyword plus signal mode maximizes specificity: the system only gets credit if it demonstrates understanding of the issue, not just mention of the topic.

There is a separate matching function in the metrics module that works differently. The finding matches discrepancy function in metrics dot py requires at least two keyword hits from the ground truth's keywords list. This is a stricter threshold than either matching mode in run evals dot py. It is used by the secondary harness for precision, recall, and F-one calculation, where you want higher confidence that a match is real.

The text extraction also handles edge cases defensively. The safe list helper returns an empty list if the value is None or not a list. The safe string helper returns an empty string if the value is None. These guards prevent type errors when the pipeline output is malformed or incomplete, which can happen when an agent fails and returns partial data.


Chapter Five: Precision, Recall, and F-one

Let us talk about the three core metrics and what each means in the legal context.

Recall measures completeness: of the eight planted errors, how many did the pipeline detect? The weighted recall formula sums the weights of detected errors and divides by the total weight of all errors. With total weight of twelve, if the pipeline catches both critical date and PPE errors (weight two each) plus the scaffolding omission (weight one), the recall is five out of twelve, or about forty-two percent.

In legal terms, recall answers: how many real problems in a brief will the system surface for a judge? Low recall means the system misses errors that a careful human reviewer would catch. For a system designed to augment judicial review, low recall reduces its value but does not actively harm anyone. The brief still gets normal human review.

Precision measures accuracy: of all the findings the pipeline produced, how many correspond to real errors? The precision evaluation uses a completely separate test case. Smith v. ABC Corp is a set of four internally consistent documents with zero planted errors. The motion's facts match the police report, the medical records, and the witness statement perfectly. If the pipeline produces findings on these clean documents, those findings are false positives.

There are four precision checks. P-01 says clean documents should produce at most one finding. Not zero, one. This allowance exists because LLM behavior is inherently noisy. A system that occasionally produces a single minor finding on clean documents is acceptable. A system that produces five findings on clean documents has a precision problem. P-02 checks that no verified facts are marked contradictory. P-03 checks that no citations are marked not supported or misleading. P-04 checks that the overall confidence score is at least zero point six. Clean documents should produce high confidence because there is nothing wrong.

In legal terms, precision answers: when the system flags something, how often is it right? Low precision means the system frequently accuses attorneys of errors they did not make. This is the more dangerous failure mode for legal AI. A system that cries wolf destroys trust. After a few false accusations, judges and law clerks stop taking the system's findings seriously, which makes them miss the real errors when they come.

F-one is the harmonic mean of precision and recall. The formula is two times precision times recall divided by precision plus recall. It is zero when either precision or recall is zero, and it equals one only when both are perfect. F-one is a balanced metric, but in this context, precision matters more than recall. The eval harness reports all three metrics separately rather than relying on F-one alone, because the asymmetric cost of errors means you need to see both components independently.

The false discovery rate is calculated as the number of false positives divided by the total number of findings. If the pipeline produces ten findings and three of them do not correspond to any ground truth error, the false discovery rate is thirty percent. This is a more intuitive metric than precision for communicating to legal stakeholders: "thirty percent of this system's findings are wrong" is more actionable than "precision is seventy percent."

There is also a combined metrics calculation that merges keyword-based matching and LLM-as-judge matching. It takes the union of matched discrepancies from both methods. If keyword matching detected errors D-01, D-02, and D-03, and the LLM judge detected D-01, D-03, and D-05, the combined set is D-01, D-02, D-03, and D-05. The combined metrics also report keyword-only matches, LLM-only matches, and matches where both methods agreed. This breakdown helps diagnose whether keyword matching or semantic matching is the bottleneck.


Chapter Six: Evidence Grounding

One of the most important checks in the eval harness is evidence grounding. This is the mechanism that prevents the pipeline from hallucinating findings that sound plausible but have no basis in the source documents.

The grounding check works by verifying that each finding's evidence text actually appears as a substring in at least one source document. The function is called calculate grounding in metrics dot py. Here is how it works step by step.

First, it loads all source documents and concatenates them into a single lowercase string. Then, for each finding, it extracts the evidence field. Evidence can be a list of strings or a list of objects with text or quote fields. For each evidence item, it normalizes the text to lowercase and checks two conditions.

The primary check is: if the normalized evidence text is at least ten characters long, does it appear as a substring in the combined source text? The ten-character minimum filters out trivial matches. Without it, single words like "the" or "and" would always match, giving a falsely high grounding rate. Ten characters is long enough to require a meaningful phrase.

The secondary check handles evidence items that are longer but composed of multiple sentences. It splits the evidence on periods and checks each fragment independently. If any fragment of at least fifteen characters appears in the source text, the finding is considered grounded. This handles cases where the LLM paraphrases the evidence slightly but includes a verbatim substring within a longer passage.

A finding is grounded if at least one of its evidence items passes either check. The grounding rate is the number of grounded findings divided by the total number of findings. The eval harness asserts that at least fifty percent of findings must be grounded.

Why fifty percent and not one hundred percent? Because some findings are based on inference rather than direct quotation. The retained control finding, for example, combines information from the police report about Donner directing the crew with the witness statement about Donner dismissing safety concerns. The synthesized finding might describe this combined insight in language that does not appear verbatim in either source. Requiring one hundred percent grounding would penalize legitimate analytical findings. Fifty percent is the threshold that catches egregious hallucination while allowing reasonable inference.

The grounding check also returns a list of ungrounded items with their index and truncated description. This makes it easy to inspect which findings lack source support and determine whether they represent genuine hallucination or legitimate inference that the algorithm cannot capture.

Evidence grounding is philosophically important because it addresses the core trust problem with LLM-based legal analysis. A judge needs to verify every claim. If the system says "the motion contradicts the police report about PPE," the judge needs to see the actual text from the police report. Grounding ensures that the system's evidence claims are traceable to specific text in specific documents. Without grounding, the system could generate plausible-sounding findings that have no basis in reality, which is exactly the failure mode that destroyed trust in AI after the Mata v. Avianca case.


Chapter Seven: LLM-as-Judge

The LLM-as-judge is the second evaluation method, complementing keyword matching with semantic understanding. It is defined in llm judge dot py and activated by setting the LLM Judge environment variable to one.

The judge uses a system prompt that says, and I will paraphrase: you are an evaluation judge for a legal brief verification system. You will compare a pipeline finding against a ground truth discrepancy and determine if they match. A match means the finding correctly identifies the same issue, even if it uses different wording. Focus on semantic equivalence, not exact keyword matching.

The user prompt then presents both the ground truth discrepancy and the pipeline finding side by side. It includes the ground truth ID, description, category, expected evidence, and expected reasoning, along with the finding's description, type, evidence, and confidence. It asks three specific questions: Does it address the same factual or legal issue? Does it identify similar evidence or contradictions? Would a legal professional consider these to be about the same problem?

The judge returns a JSON object with three fields: match (a boolean), confidence (a float from zero to one), and reasoning (a string explanation). A finding is considered a match only if both the match field is true and the confidence is at least zero point five. This dual threshold prevents weak matches from inflating the recall score.

The judge all function iterates over all findings and all ground truth items. For each finding, it checks against each unmatched ground truth item. Once a ground truth item is matched, it is removed from consideration to prevent double-counting. This greedy matching means the order of findings can affect which ground truth items get matched, but in practice with eight ground truth items and typically six to ten findings, this is not a significant concern.

Why is the LLM judge opt-in rather than the default? Three reasons.

First, determinism. Keyword matching produces the same result every time given the same input. The LLM judge can produce different results on different runs because of temperature, prompt sensitivity, and API-level non-determinism. An evaluation system should be the most stable component. You want to know that a change in scores reflects a real change in the pipeline, not random variation in the evaluator.

Second, cost and latency. The LLM judge makes one API call for every finding-ground-truth pair. With ten findings and eight ground truth items, that is up to eighty API calls per evaluation run. Keyword matching is free and instantaneous.

Third, the eval-of-eval problem. When you use an LLM to evaluate an LLM, you need to evaluate the evaluator. How do you know the judge is accurate? You would need ground truth for the judge's decisions, which creates infinite regress. Keyword matching sidesteps this problem because its behavior is transparent and verifiable by inspection.

That said, the LLM judge catches things keyword matching misses. The example given in the existing interview prep is that "date discrepancy" and "incorrect incident date" mean the same thing but share no keywords. The judge recognizes semantic equivalence. This is why the combined metrics use the union of both methods: you get the determinism and transparency of keywords plus the semantic coverage of the judge.

The temperature for the judge is set to zero point zero, which maximizes determinism. The result parsing strips markdown code fences and whitespace before JSON parsing. If the judge call fails for any reason, the match defaults to false with zero confidence and the error message is captured in the reasoning field. This means judge failures are recorded as data rather than exceptions.


Chapter Eight: The Six Eval Categories

The Python eval harness in run evals dot py organizes its thirty-one assertions into six categories. Each category tests a different dimension of pipeline quality. Let us walk through all six in detail.

Category one is recall. This has eight checks, one per planted error. Each check calls the check ground truth function with the pipeline report and the ground truth definition. The function extracts text from the specified report sections, concatenates it, and applies the matching algorithm in the appropriate mode: any or keyword plus signal. Each recall check inherits the weight from its ground truth definition, so critical errors contribute more to the aggregate recall score.

The recall score is calculated as the sum of weights for passed checks divided by the sum of all weights. With four errors at weight two and four at weight one, the total weight is twelve. A perfect score requires detecting all eight errors.

Category two is precision. This has four checks run against the clean document report, not the Rivera case. P-01 checks that clean documents produce at most one top finding. P-02 checks that no verified facts have contradictory status. P-03 checks that no citations are marked not supported or misleading. P-04 checks that the overall confidence score is at least zero point six.

The precision score is the number of passed checks divided by four. These are binary pass-fail checks, not weighted, because all four precision properties are equally important. A system that produces zero false findings but has low confidence on clean documents still has a precision problem: it is expressing uncertainty about things that are clearly correct.

Category three is hallucination. This has four checks. H-01 checks for fabricated case names. It serializes all verified citations to JSON, lowercases it, and searches for six known fabricated names: Smith v. Jones, Garcia v. Acme, Wilson v. BuildRight, Johnson v. Safety, Martinez v. Construct, and Brown v. Steel. These were chosen because they follow common patterns that LLMs use when hallucinating case names. H-02 checks that at least two known case names from the actual MSJ appear in the extracted citations. The known case names are Privette, Seabright, Whitmore, Kellerman, Dixon, Okafor, Rivera, and Harmon. This ensures the parser is extracting real citations, not inventing them. H-03 checks that at least fifty percent of findings are grounded with evidence, where "grounded" means at least one evidence item has more than ten characters of substantive content. H-04 checks that no findings reference document types not in the case, specifically depositions, expert reports, surveillance footage, and tax returns. If the system mentions a deposition when no deposition exists in the case files, it is hallucinating a source.

The hallucination rate is the number of failed checks divided by four. Zero is ideal.

Category four is cross-document consistency. This has five checks. C-01 verifies that at least one verified fact references multiple source documents, either through contradictory sources or by having more than one supporting source. C-02 checks that at least one fact references the police report. C-03 checks for witness statement references. C-04 checks for medical records references. C-05 verifies that the metadata confirms all four document types were analyzed.

These five checks ensure the pipeline is actually performing cross-document verification, not just analyzing the motion in isolation. The whole point of the BS Detector is comparing the brief's claims against supporting documents. If the pipeline does not reference the police report, it is not doing its job, even if it accidentally produces correct findings.

Category five is uncertainty expression. This has four checks. U-01 verifies that at least one citation has a confidence value below one point zero. A system that assigns perfect confidence to every citation is overconfident, because citation verification inherently involves uncertainty when backed by LLM knowledge rather than a legal database. U-02 checks that the three main confidence scores, citation verification, fact consistency, and overall, are not all one point zero and not all zero point zero. All ones means the system is not expressing any uncertainty. All zeros means it is not expressing any confidence. Either extreme is pathological. U-03 checks that the unknown issues list exists as an array. This field is where the pipeline reports things it could not verify, and its existence signals that the system has a mechanism for expressing uncertainty. U-04 checks that no citation that has been flagged as not supported or misleading claims one hundred percent confidence. A system that says "this citation is misleading, confidence one point zero" is contradicting itself: if the citation is problematic, there should be some epistemic humility about that assessment.

Category six is structure. This has six checks. S-01 verifies that the report contains all required top-level fields: motion ID, verified citations, verified facts, top findings, and confidence scores. S-02 checks that citations are structured objects with a citation sub-object, not raw text strings. S-03 makes the same check for facts. S-04 verifies that confidence scores are valid numbers between zero and one. S-05 checks that the judicial memo is a structured dictionary with a memo field of at least fifty characters. S-06 verifies that finding IDs are present and unique.

The structure checks might seem mundane, but they are critical for a multi-agent system. Each agent passes data forward as serialized Pydantic models. If the citation verifier returns raw strings instead of structured citation objects, the downstream synthesizer cannot extract the fields it needs. Structure checks catch data passing bugs that would otherwise manifest as confusing failures in later pipeline stages.


Chapter Nine: Promptfoo Integration

The second evaluation layer is the promptfoo configuration in promptfoo config dot yaml. This file defines twenty assertions that run against the same pipeline output as the Python harness. The integration between the two layers is mediated by a Python provider and a file-based cache.

The provider is defined in evals slash provider dot py. When promptfoo invokes a test case, it calls the call API function. This function checks the mode variable from the test context. If the mode is "clean," it gets the clean document report. Otherwise, it gets the default Rivera case report. The function returns the report wrapped in a dictionary with an output key containing a report key.

The critical design decision in the provider is the caching strategy. The pipeline takes twenty-five to thirty seconds to run. Promptfoo runs tests in parallel across multiple workers. Without caching, each worker would trigger a separate pipeline run, wasting tokens and time.

The caching uses three mechanisms. First, a file-based cache. The report is saved as JSON to a dot report cache dot json file with a one-hour time-to-live. When a worker needs a report, it first checks if the cache file exists and is less than one hour old. If so, it reads the cached report directly.

Second, file locking using fcntl. If the cache is missing or stale, the worker acquires an exclusive lock on a lock file before running the pipeline. After acquiring the lock, it double-checks the cache. This double-check pattern handles the race condition where another worker already ran the pipeline while this worker was waiting for the lock. If the cache is now fresh, it reads it and releases the lock. If the cache is still stale, it runs the pipeline, writes the cache, and releases the lock.

Third, cache warming. The Python harness in run evals dot py runs the pipeline first and saves the report to the same cache file before invoking promptfoo. This means promptfoo workers almost always hit a warm cache. The lock-based cache miss path is a safety net, not the normal execution path.

The one-hour TTL balances two concerns. Too short, and the cache expires between the Python harness and the promptfoo run. Too long, and stale results from a previous evaluation contaminate the current run. One hour is long enough for a full eval cycle but short enough that the next evaluation starts fresh.

The promptfoo assertions are written in JavaScript and grouped into four categories. The structural checks mirror the Python harness structure checks: required fields, minimum citation count of three, minimum fact count of three, valid confidence scores between zero and one, structured judicial memo, and four documents analyzed. The discrepancy detection checks cover all eight planted errors using the same logic as the Python harness but implemented in JavaScript with the same dual-mode matching. The quality checks verify no fabricated case names, at least one high-confidence finding, at least one contradictory fact, at least one flagged citation, and evidence grounding at fifty percent or above. The negative test sends clean documents through the pipeline and checks for at most one finding, zero contradictory facts, and zero bad citations.

Each discrepancy detection check in promptfoo uses an immediately invoked function expression, or IIFE, to keep the JavaScript self-contained. These functions extract text from top findings and verified facts, filtering verified facts to only include those with contradictory, partial, or could not verify status. This filtering is important: a verified fact with consistent status should not count as a detection of an error.

The D-01 date check, for example, extracts description text from top findings and summary plus fact text from filtered verified facts, lowercases everything, and returns true if any text contains "march twelve," "march fourteen," or "date discrepancy." The D-02 PPE check adds the signal word requirement, matching the keyword plus signal mode from the Python harness.

Some promptfoo assertions have weight annotations. The critical discrepancy checks, D-01, D-02, D-03, and D-05, have weight two. The standard checks have weight one. These weights affect the aggregate promptfoo score displayed in the web viewer.


Chapter Ten: Agent-Level Evals

The third evaluation layer isolates individual agents for focused testing. This is where you diagnose whether a failure is in a specific agent's prompt or in the pipeline's data flow.

The citation extraction eval in citation extraction dot yaml tests the Document Parser Agent. It sends MSJ text directly to the citation extraction prompt and checks the output. There are two test cases. The first sends a paragraph containing the Privette citation with the fabricated "never" quote, plus the Dixon and Okafor out-of-state citations. It asserts four things: the output is valid JSON, the extracted citations include Privette, the extraction captures the "never" quote in context, and it finds both Dixon and Okafor. The second test case is a hallucination check. It sends a paragraph containing only the Kellerman citation and asserts that the extraction finds Kellerman but does not hallucinate Privette or Dixon, which were not in the input text.

The fact checking eval in fact checking dot yaml tests the Fact Checker Agent with six test cases, one for each of the ground truth fact errors: date, PPE, control, scaffolding, post-incident spoliation, and statute of limitations. Each test case provides the MSJ claims, police report text, medical records text, and witness statement text as separate variables. The assertions use both JavaScript checks and LLM rubric checks.

The JavaScript checks verify structural correctness: valid JSON output and that the verified facts array contains at least one fact with the expected keywords and contradictory status. The LLM rubric checks use semantic evaluation to verify that the response identifies the correct issue. For example, the date check's rubric says the response must identify the date discrepancy where the MSJ states March fourteenth but all other documents state March twelfth, and this should be marked contradictory.

Using both JavaScript and LLM rubric assertions in agent-level evals gives you the best of both worlds. The JavaScript checks are deterministic and fast, catching structural failures immediately. The LLM rubric checks catch cases where the output is structurally correct but semantically wrong, like a fact checker that marks a date as contradictory but gives the wrong reasoning.

Each agent-level test uses the DeepSeek API directly, with temperature set to zero point one for generation and zero point zero for the evaluation judge. The output transform strips markdown code fences, which is necessary because DeepSeek sometimes wraps JSON output in triple backtick blocks even when the prompt says not to.

The agent-level evals are valuable for three reasons. First, they provide faster iteration. Testing a single agent takes three to five seconds. Testing the full pipeline takes twenty-five to thirty seconds. When you are tuning a prompt, that five-to-one speed difference compounds over dozens of iterations. Second, they isolate root causes. If the pipeline misses the date error, is it because the fact checker failed to detect it, or because the synthesizer dropped it when ranking findings? Agent-level evals answer that question directly. Third, they document expected behavior. Each test case serves as a specification: given these inputs, this agent should produce this output. New team members can read the tests to understand what each agent is supposed to do.


Chapter Eleven: Prompt Precision A/B Tests

The fourth evaluation layer quantifies the impact of prompt engineering through controlled A/B experiments. There are two A/B configs: one for citation verification and one for fact checking.

The citation verification A/B test in prompt precision citations dot yaml runs the same four test cases against two prompts. The precise prompt is nineteen lines long and includes four specific verification instructions, an important legal knowledge block naming Privette and Seabright with their actual holdings, an explicit schema definition with the four status enum values, and the jurisdiction specified as California state court. The imprecise prompt is seven lines long and simply says "review the citation and tell me if it is valid, return findings as JSON."

The test cases cover four scenarios. CIT-001 is the Privette "never" misquote. The precise prompt should flag this because its legal knowledge block explicitly states that the word "never" does not appear in the Privette holding. The imprecise prompt must rely on the LLM's parametric knowledge, which may or may not know this detail. CIT-002a is Dixon, the Texas case. The precise prompt should flag this because it explicitly says to flag out-of-state cases and specifies the jurisdiction as California state court. The imprecise prompt has no jurisdiction context. The precision test case presents an accurate Privette claim, one that correctly describes the rebuttable presumption without inserting "never." Both prompts should mark this as supported, but the imprecise prompt may false-flag it because it lacks the nuance to distinguish the accurate characterization from the misquotation. The uncertainty test uses a fictitious citation, Marchetti v. Western Bay Developers. Both prompts should express doubt, but the precise prompt's explicit "could not verify" status in the schema guides the model toward hedging rather than fabricating a verdict on a nonexistent case.

Each test case has schema checks (valid JSON, correct field names, enum compliance), recall checks (correct status assignment), and LLM rubric checks (semantic accuracy of the reasoning). The scores are tracked separately for each prompt, so you can compare precise versus imprecise on every metric.

The fact checking A/B test in prompt precision facts dot yaml runs three test cases against the precise and imprecise fact-checking prompts. The precise prompt is twenty-six lines with six numbered checking categories and an explicit schema definition for the verified facts array. The imprecise prompt is fifteen lines that just say "compare these documents and find inconsistencies, return findings as JSON."

The key design insight of the A/B tests is that they measure the causal impact of prompt specificity, not just correlation. Because both prompts receive identical inputs and are evaluated with identical assertions, any difference in scores is attributable to the prompt difference alone. The precise prompt's advantage comes from three additions: explicit checking categories that force the model to evaluate each dimension, domain knowledge that provides facts the model might not have in its training data, and schema definitions that constrain the output space to valid enum values.

The measurable impact is significant. The precise citation verification prompt consistently outperforms the imprecise one on the Privette misquotation detection because of the legal knowledge block. It outperforms on jurisdiction flagging because of the explicit California state court context. And it outperforms on the fictitious citation because the schema's "could not verify" option gives the model a way to express uncertainty without fabricating a holding.


Chapter Twelve: The Reflection Honesty Eval

The reflection honesty eval is architecturally unique in the eval harness because it evaluates a human-written document, not pipeline output. It assesses whether the project's reflection document is genuinely honest rather than performatively humble.

The config in reflection honesty dot yaml uses a custom Python provider that reads the reflection markdown file directly. There is no LLM generation cost for the "output" because the reflection already exists. The evaluation judge is DeepSeek R-one, which is a reasoning model. The choice of a reasoning model for judging is deliberate: assessing honesty requires nuanced reasoning about intent, not just pattern matching.

The eval has six dimensions, each with a weight.

Real weaknesses, weight three. This is the most important dimension. The rubric requires the reflection to name specific technical failures or engineering shortcomings, not just time constraints or scope limitations. Concrete issues like incorrect algorithms, missing edge cases, or architectural flaws count. Generic statements like "I ran out of time" do not count. The reflection must describe at least two genuine technical weaknesses with enough detail to reproduce or verify them.

Qualified numbers, weight two. Any quantitative claim must include appropriate qualification or context. For example, saying "cuts roughly forty percent off execution time" should indicate how that was measured or acknowledge it as an estimate. Unqualified precise numbers presented as fact without methodology score fail. If no quantitative claims are made, it scores pass by default.

Eval self-critique, weight two. The reflection must acknowledge limitations of its own evaluation methodology. Not just the system under test, but the evaluation itself. Look for honest assessment of eval coverage gaps, metric weaknesses, dataset size limitations, or potential biases. Simply listing what the eval tests is not self-critique. It must identify at least one concrete way the evaluation could give misleading results.

Scope honesty, weight one. The reflection must clearly distinguish between features that are fully completed, partially implemented, and deliberately skipped. Vague language that obscures completion status scores fail.

Future specificity, weight one. Any future work suggestions must be tied to specific observed failures or limitations documented elsewhere in the reflection. Generic improvement ideas like "add more tests" that are not grounded in a described problem score fail.

Not performative, weight one. The reflection must avoid performative humility, which is false modesty that signals awareness of best practices without actually engaging with real problems. Patterns like "of course this is just a demo" without substantive analysis score fail. Self-critical statements must be backed by specific evidence.

The composite honesty score is calculated as a weighted sum divided by ten: real weaknesses times three, plus qualified numbers times two, plus eval self-critique times two, plus scope honesty, plus future specificity, plus not performative, all divided by ten. This produces a score between zero and one.

This eval is meta in the best sense. It evaluates whether the developer was honest about the system's limitations. In the context of legal AI, this matters enormously. A developer who cannot honestly assess their own system's weaknesses is dangerous in a domain where overconfidence has real consequences. The reflection honesty eval encodes the principle that intellectual honesty is a measurable engineering quality, not just a personal virtue.


Chapter Thirteen: Persistence and Trend Tracking

The eval harness includes a SQLite persistence layer defined in evals slash db dot py. Every evaluation run is saved to a database, enabling trend tracking across commits.

The database has a single table called eval runs with thirteen columns. The run ID is a UUID hex string serving as the primary key. The timestamp is an ISO eight six oh one string in UTC. The git SHA is captured by running git rev-parse short HEAD as a subprocess. This ties each eval run to a specific code version.

Then there are the scalar metric columns: precision, recall, F-one score, hallucination rate, true positives, false positives, and false negatives. These are stored as individual columns for efficient querying. You can query "show me all runs where recall dropped below seventy percent" without deserializing JSON.

Finally, there are three JSON text columns: metrics (the full metrics dictionary), findings (the list of extracted findings), and report (the complete pipeline report). These provide full detail for post-hoc analysis. The metrics JSON is redundant with the scalar columns, but having both enables quick queries on scalar columns and deep inspection via JSON.

The init db function creates the table with CREATE TABLE IF NOT EXISTS, making it safe to call multiple times. It also enables Write-Ahead Logging mode, or WAL, which allows concurrent reads while a write is in progress. This matters because the promptfoo provider might be reading the database while the Python harness is writing to it.

The save run function calls init db internally, guaranteeing the table exists before any insert. This defensive pattern means the database is self-bootstrapping. You never need to run a migration or setup script. The first call to save run creates both the database file and the table.

The get runs function retrieves the most recent runs ordered by timestamp. It uses sqlite3 dot Row as the row factory for dictionary-like access, and deserializes the JSON columns back to Python objects.

The git SHA correlation is the key feature for regression detection. When a prompt change causes recall to drop, you can identify exactly which commit introduced the regression. The workflow is: run evals, commit, run evals, commit. Each eval run is tagged with the commit SHA. If you plot recall over time, drops correlate with specific commits, and you can inspect those diffs to understand what changed.

The persistence layer stores both the harness output (from evals slash harness dot py) and the full eval suite output (from run evals dot py). The harness stores keyword metrics, grounding data, and optionally LLM judge metrics and combined metrics. The full eval suite stores the six-category results and the aggregate scores.

A design limitation worth noting: the persistence layer does not currently store promptfoo results. Promptfoo has its own storage and web viewer. This means the SQLite database captures Python harness metrics but not the promptfoo assertion results. To get a complete picture, you need to look at both the SQLite database and the promptfoo dashboard.


Chapter Fourteen: What is Missing and What Breaks

No evaluation system is perfect, and intellectual honesty requires acknowledging the gaps. Let us walk through what is missing from the eval harness and where it would break under real-world conditions.

The most significant limitation is single test case coverage. The entire eval harness runs against one case: Rivera v. Harmon. Eight planted errors in one motion for summary judgment. This is enough to validate the pipeline's architecture and catch regressions, but it tells you almost nothing about generalization. Does the system work on a motion for summary judgment in a patent case? What about a habeas corpus petition? A family law motion? Legal documents vary enormously in structure, vocabulary, and complexity. Evaluating on a single case is like evaluating a spam filter on one email.

The fix is obvious: more test cases. But creating high-quality legal test cases with carefully planted errors requires legal domain expertise. Each case needs a realistic motion, matching supporting documents, and errors that are subtle enough to be meaningful. This is a content creation bottleneck, not a technical one.

Second, the eval-of-eval problem. The keyword matching algorithm itself has not been formally validated. We assume that if the text contains "march twelve" and "march fourteen," the system has detected the date error. But what if the system mentions both dates while concluding that the dates are consistent? The keyword match would pass, but the system's analysis would be wrong. The keyword plus signal mode partially addresses this by requiring both a topic keyword and a signal word, but it is not a complete solution. A truly robust evaluation would need to parse the system's reasoning, not just its vocabulary.

Third, the grounding check has a fundamental limitation: it only checks whether evidence text appears as a substring in source documents. It does not check whether the evidence is used correctly. A finding could quote the police report accurately but draw the wrong conclusion from it. The grounding check would pass because the quote is real, but the finding would still be wrong. Grounding checks prevent fabrication but do not verify reasoning.

Fourth, the clean document precision test is simplistic. Real-world precision failures are more nuanced than false positives on perfectly clean documents. A more realistic precision test would use documents with minor inconsistencies that are not legally significant. For example, two documents might use different formatting for the same date, or a witness statement might use slightly different phrasing than the police report. These are normal variations that should not be flagged. Testing precision only on perfectly consistent documents misses this category of false positives.

Fifth, the LLM-as-judge has not been calibrated. The confidence threshold of zero point five was chosen as a reasonable default, not derived from empirical analysis. A proper calibration would require running the judge on a large set of labeled finding-ground truth pairs and finding the threshold that maximizes agreement with human labels. Without calibration, the zero point five threshold might be too permissive, counting weak semantic matches as detections, or too strict, missing valid matches.

Sixth, there is no evaluation of the judicial memo agent. The structure checks verify that the memo is a dictionary with a memo field of at least fifty characters, but there is no quality check on the memo's content. Is the legal reasoning sound? Is the language appropriate for a judicial audience? Are the recommended actions reasonable? These are subjective qualities that would require either expert human evaluation or a carefully designed LLM rubric.

Seventh, the eval does not measure latency or cost. A system that catches eight of eight errors in ten minutes is less useful than one that catches seven of eight in thirty seconds. Production evaluation should include performance budgets alongside accuracy metrics.

Eighth, there is a dataset bias concern. The eight planted errors were designed by the same person who wrote the prompts and the matching algorithm. This creates a risk of overfitting: the prompts may be tuned to catch exactly these eight error patterns while missing others. An independent test set created by someone unfamiliar with the prompts would provide a more honest assessment of generalization.

Despite these limitations, the eval harness provides a solid foundation for iterative improvement. Each limitation has a clear fix. More test cases improve coverage. Human evaluation of a labeled dataset calibrates the LLM judge. Adversarial testing with near-miss documents improves precision measurement. The architecture supports all of these extensions without structural changes. The gaps are in data and calibration, not in the evaluation framework itself.

And that is perhaps the most important meta-lesson. An eval harness is never finished. It is a living system that grows with the pipeline it evaluates. The fact that you can articulate what is missing is itself evidence that the evaluation framework provides useful signal. A system with no evaluation cannot even tell you what it does not know.

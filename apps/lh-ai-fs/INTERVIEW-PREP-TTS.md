Chapter Two: Architecture Walkthrough

The BS Detector is a five-agent pipeline built in Python with FastAPI. Five specialized AI agents process legal documents in a defined sequence with one parallelization point. Here is the flow: documents go into the Document Parser, which extracts citations and facts. Then the Citation Verifier and the Fact Checker run in parallel because they are independent tasks. After both complete, the Report Synthesizer aggregates and ranks findings. Finally, the Judicial Memo Agent writes a formal memo for the judge.

The orchestrator that coordinates all of this is called Pipeline Orchestrator, and here is the critical detail: it makes zero LLM calls. Its behavior is fully deterministic. Only the agents introduce non-determinism through their LLM interactions. This separation matters because it means pipeline control flow can be reasoned about, tested, and debugged independently of LLM behavior. If the orchestrator itself used an LLM to decide which agents to invoke or how to route findings, the pipeline's behavior would become unpredictable. A deterministic orchestrator means that given the same agent outputs, the pipeline always produces the same final report. This makes evaluation reproducible and debugging tractable.

Why five agents and not three or seven? The answer comes down to cognitive task separation. Each agent has a distinct job. The parser extracts, and its instruction is "be comprehensive, find everything." The citation verifier and fact checker analyze, and their instruction is "be skeptical." The synthesizer aggregates and ranks. The memo writer communicates in legal language. With fewer agents, you conflate these instructions. When you ask an LLM to simultaneously extract citations and verify them, it tends to skip citations it considers unimportant. But unimportant citations might be the ones most worth checking, like the out-of-state citations buried in a footnote. With more agents, say seven or eight, you add coordination overhead and additional LLM calls without meaningful specialization gains. The five-agent decomposition is the Pareto optimum.

The empirical evidence backs this up. The single-prompt version of this system, where everything was done in one LLM call, caught three to four of eight planted errors and often hallucinated findings. The multi-agent version catches six to eight. The architecture change, not prompt tuning, drove the improvement.

All inter-agent communication uses typed Pydantic contracts. Every stage passes data forward as Python dictionaries serialized from Pydantic models using model dump. This is deliberate. Three reasons. First, JSON serializability: intermediate data can be logged, cached, or transmitted over the network. Second, loose coupling: agents do not import each other's Pydantic models, so the parser does not need to know about Verified Citation. Third, future distributability: if agents were moved to separate services, the dictionary-based interface requires no changes. The status enums constrain the output space. For citations: supported, not supported, could not verify, or misleading. For facts: consistent, contradictory, partial, or could not verify. The status map dictionary maps whatever strings the LLM returns to these enum values, with a safe default to could not verify for anything unmapped. This is defensive programming against LLM output variability.

The LLM service supports DeepSeek for cloud and Ollama with Qwen two point five seven B for local development, both through LangChain's Chat OpenAI abstraction. DeepSeek was an assignment constraint. The architecture abstracts the model behind LLM Service, so swapping to GPT-4 or Claude requires changing an environment variable. But prompt engineering is model-specific, so changing models requires re-running evaluation and potentially re-tuning prompts.


Chapter Three: Parallel Execution and Error Handling

There are two levels of parallelism in the pipeline. The first is at the orchestrator level, at line 110 through 112 of orchestrator.py. The citation verifier and fact checker run concurrently using asyncio.gather. This exploits the fact that these two tasks are completely independent. Verifying whether a legal citation is real has nothing to do with checking whether the brief's facts match the police report. Running them sequentially would waste time. The parallel topology cuts wall-clock time by roughly forty percent for the verification stage.

The second level of parallelism is inside the citation verifier itself, at lines 72 through 75 of citation verifier.py. Each individual citation is verified concurrently. With five to ten citations in a typical motion for summary judgment, sequential verification would take five times three to ten seconds, so fifteen to fifty seconds total. Parallel verification completes in the time of the slowest single citation, three to ten seconds. The citations are independent. Verifying Privette v. Superior Court does not require knowing anything about Dixon v. Lone Star.

Both gather calls use return exceptions equals true. This is essential. Without that flag, if one citation verification fails, all in-flight verifications would be cancelled. In legal verification, getting four out of five results is strictly better than getting zero out of five. The flag ensures that each task completes independently, and failures are captured as data rather than propagated as exceptions.

Error handling uses a three-layer isolation architecture. The first layer is the inner try-except inside the verify one method. This catches per-citation LLM failures and returns a Verified Citation with could not verify status and error details in the notes field. Individual citation failures are recorded as data, not as exceptions.

The second layer is the return exceptions equals true in the asyncio.gather call. This catches any exceptions that escape the inner try-except, which should not happen but might in edge cases like serialization errors. Failed items are replaced with error Verified Citation entries.

The third layer is at the orchestrator level. If the entire citation verifier agent raises an exception, the orchestrator catches it and continues with an empty citation results list. The pipeline keeps running.

Why three layers instead of one? Each layer handles a different failure scope. Layer one handles individual citation failures, which are common because some citations are harder to verify. Layer two handles aggregate failures, which are rare. Layer three handles agent-level crashes, which are very rare. One layer could not handle all three scopes without becoming a complex conditional mess.

The graceful degradation hierarchy has five levels. Level zero is a full report where all five agents succeed. Level one is a report without the judicial memo, because the memo agent failed, but you still have citations, facts, findings, and confidence scores. Level two is a report with partial verification because one of the citation or fact agents failed, but you still have results from the healthy agent. Level three is a report with only raw data because the synthesizer failed, so you have verified citations and facts but no ranked findings or scores. Level four is an error report because the parser failed. Without citations, the citation verifier has nothing to verify, and running only the fact checker would produce a misleadingly partial result.

Every agent transition is tracked with millisecond-precision timing using four helper functions in the orchestrator: track, start, succeed, and fail. This is a lightweight distributed tracing system. The Agent Status entries in the final verification report make failures transparent. A black box that says "analysis complete" without showing which components succeeded is unacceptable for judicial AI. Judges need to know what the system could and could not do.


Chapter Four: Agent Deep Dives

Let's walk through each agent in detail.

The Document Parser Agent makes a single LLM call with a citation extraction prompt, asking the model to extract citation text, claimed proposition, source location, and context for every citation in the motion for summary judgment. Output is validated against an Extraction Result Pydantic model. The key design decision: the parser's extraction is context-free. The prompt has no case context placeholder. This is intentional. You want to extract all citations, including the correct ones. Injecting case-specific knowledge at the extraction stage would bias the parser toward flagging certain citations, which is the verifier's job, not the parser's. A known limitation: the parser only processes the motion for summary judgment, not the supporting documents. This means the fact checker must perform both extraction and verification in a single pass, which is a heavier cognitive load for the LLM.

The Citation Verifier Agent implements per-citation parallelism. Each citation gets its own LLM call through the verify one method, which formats the verification prompt with the citation text, claimed proposition, context, and case-specific knowledge. The status mapping design uses a dictionary that maps LLM-returned strings to enum values, with a fallback to could not verify for any unmapped status. This handles cases where the LLM returns unexpected strings like "partially supported" or "uncertain." No matter what the model returns, the system produces a valid enum value.

The Fact Checker Agent makes a single LLM call with the most complex prompt in the system: sixty-seven lines. The prompt defines eight checking categories: date consistency, PPE safety, work control, scaffolding condition, OSHA compliance, injury details, statute of limitations, and strategic omission. Why a single prompt instead of eight separate prompts for each category? Eight prompts would cost eight times the tokens and time. A single prompt can see all categories simultaneously, which matters because categories interact. The wrong date affects the statute of limitations calculation. The trade-off is prompt complexity.

The precision rules in the prompt are critical. There are seven of them, each addressing a specific false-positive pattern discovered during development. Rule one: contradictory means the motion makes a specific claim that is directly refuted. Without this, the LLM flagged implicit claims. Rule two: an omission is only contradictory if the motion actively hides material evidence. Without this, every fact in the police report that was not mentioned in the motion got flagged. Rule six is the meta-rule: when in doubt, prefer consistent or could not verify over contradictory. This encodes the precision-over-recall trade-off. For legal AI, false positives are more damaging than false negatives because false positives erode judicial trust in the system. Rule seven: skip any category that does not apply. Without this, the LLM would generate entries for all eight categories regardless of relevance.

Why are these rules in the prompt rather than in post-processing code? Because the precision rules need to influence the LLM's reasoning during generation, not filter its output afterward. An LLM that generates a contradictory finding and then has it filtered out has already wasted tokens and processing time. An LLM that never generates the finding in the first place is cheaper and faster.

The Report Synthesizer Agent takes citation and fact verification results, truncated to ten thousand characters each, and produces three outputs: top findings ranked by severity, confidence scores, and unknown issues. The synthesizer's job is editorial. Raw verification results are noisy. The citation verifier might produce ten entries, seven of which are supported. Those are not findings, they are confirmations. A judge does not want to read eighteen entries to find six actual problems. The synthesizer surfaces what matters and suppresses what is noise.

The Judicial Memo Agent is separate from the synthesizer because synthesis and memo writing are different skills. The synthesizer does analytical work: ranking and scoring. The memo agent does persuasive legal writing. An LLM prompted for analysis produces better analysis when it is not simultaneously trying to write a polished memo. The memo returns four structured fields: memo text, key issues, recommended actions, and overall assessment. This enables the UI to display each section separately rather than requiring the frontend to parse natural language.


Chapter Five: Citation Verification, the Key Topic

This is the topic they will probe hardest. Lead with honesty.

For the submission, citation verification relies on the LLM's training data for case existence and holding accuracy. This is the biggest gap for production. The case context file is nineteen lines of hardcoded knowledge about the Rivera v. Harmon test case. For unknown cases, the system asks DeepSeek whether it has heard of the case, which is fundamentally different from querying a legal database.

Why does this gap matter so much? A fabricated citation is worse than a null pointer. It can result in sanctions, case dismissal, loss of a bar license. Judges already distrust AI partly because of hallucinated citations. The landmark case is Mata v. Avianca from 2023, where attorneys used ChatGPT to draft a motion and it generated six entirely fabricated case citations with fake quotations and internal cross-references. The attorneys were fined five thousand dollars. Since then, courts have become less tolerant, not more. In the Crabill case in Colorado, an attorney received a one-year suspension, the first AI-related attorney suspension in the United States. In Park v. Kim before the Second Circuit, an attorney was referred to the court's Grievance Panel. The escalation pattern is clear.

This is exactly why Learned Hand built a closed universe. The answer to hallucination is constrained retrieval, not better prompts. And here is how you should frame your system's gap: the verification logic is right, but the knowledge backend needs to be replaced. The case context injection pattern, where domain knowledge is injected into the verification prompt, is architecturally correct. In production, the get case context function would query a legal database instead of returning a hardcoded string. The interface is right. The implementation is a placeholder.

Now articulate the production vision. A five-tier citation verification model. Tier one is existence. Query a case law database, verify the volume, reporter, and page number, confirm the case name matches. This is a deterministic lookup, no LLM judgment needed. CourtListener is free and open-source, covers all federal cases, and has an API. Westlaw and LexisNexis provide comprehensive state coverage.

Tier two is holding verification. Use retrieval-augmented generation to fetch the actual opinion text, then compare the brief's characterization against the real holding. The LLM's role shifts from "do you know this case," which is unreliable, to "does this retrieved passage support the brief's claim," which is much more reliable because the LLM is reading the actual text, not recalling from memory.

Tier three is jurisdiction. Parse the reporter abbreviation to determine the jurisdiction. S.W.3d means Texas. So.3d means Florida. Cal.4th means California. This is a rule-based lookup table, no LLM needed. Flag non-binding authority when a Texas case is cited in a California motion.

Tier four is treatment. Run a Shepard's or KeyCite check to determine whether the case has been overruled, reversed, questioned, or distinguished. A citation to an overruled case is the most dangerous kind of error because it is a real case the attorney may have found through legitimate research, but the law has changed.

Tier five is proposition-level verification. Extract the specific legal question the brief claims the case answers, search the opinion for relevant passages. This catches the subtlest error: a real case, correctly cited, but mischaracterized. The brief says Privette holds that defendants are "never liable," but the actual doctrine has exceptions for retained control and concealed hazards. That word "never" was inserted by the brief's author.

The current system has the logic for tiers one through three but backed by LLM knowledge instead of a database. The architecture does not change. Only the knowledge source changes. And the could not verify status is the honest middle ground. Without it, the model would be forced to choose between supported, which might be wrong, and not supported, which might be unfair to the attorney. In production with a legal database, items that return could not verify would genuinely indicate a problem. From an LLM, could not verify just means "I do not have enough information."


Chapter Six: The Eval Harness, Your Standout

Vienna called the eval harness the standout. Be ready to go deep.

The evaluation system has five layers. Layer one is keyword matching. The finding matches discrepancy function requires at least two keyword hits from the ground truth. It also calculates precision, recall, F1 score, and false discovery rate. And there is a grounding check that verifies evidence is traceable to source documents using a ten-character minimum substring match.

Layer two is LLM-as-judge. This is opt-in, activated by setting the LLM Judge environment variable to one. It uses semantic matching with a confidence threshold of zero point five. The prompt asks: does this pipeline finding correctly identify the ground truth discrepancy?

Layer three combines both methods. It takes the union of keyword and LLM matches and shows keyword-only, LLM-only, and both-agreed categories. Why both methods? Keyword matching is deterministic, fast, and transparent. LLM-as-judge catches semantic equivalences that keyword matching misses. For example, "date discrepancy" and "incorrect incident date" mean the same thing but share no keywords. The union gives best coverage.

Layer four is the comprehensive evaluation with thirty-one assertions across six categories. Eight recall checks, one per planted error, with critical errors getting double weight. Four precision checks ensuring clean documents produce at most one finding, zero contradictions, no bad citations, and high confidence. Four hallucination checks verifying no fabricated case names, known case names present, at least fifty percent of findings grounded, and no invented document types. Five cross-document consistency checks. Four uncertainty checks. And six structure checks verifying required fields, valid confidence floats, structured memo, and unique finding IDs.

Layer five is persistence. A SQLite database stores every evaluation run with metrics, findings, and the full report. The git SHA is recorded for each run, enabling trend tracking across commits.

The ground truth is Rivera v. Harmon with eight planted errors. D-01 is a date error: the motion says March fourteenth but every other document says March twelfth. D-02 is a PPE error: the motion says "no PPE" but the evidence shows harness, hat, and vest. D-03 is the Privette misquotation: the word "never" was inserted to change the doctrine's meaning. D-04 is a statute of limitations error using the wrong incident date. D-05 is a control error: the motion says "no control" but Foreman Donner directed the work. D-06 is a jurisdiction error: Dixon is a Texas case and Okafor is a Florida case, neither binding in California. D-07 is a scaffolding omission: the motion omits rust and plywood defects. D-08 is a spoliation finding: the post-incident scaffold rebuild constitutes destruction of evidence.

The matching algorithm uses two modes. The "any" mode triggers on any keyword match, used for errors where a single keyword is specific enough. The "keyword plus signal" mode requires both a keyword and a signal word, used for errors where the keyword alone is ambiguous. This dual-mode approach prevents false matches while maintaining sensitivity.

The precision test uses clean documents under the name Smith v. ABC Corp. All four documents are internally consistent with no planted errors. The system should produce at most one finding, zero contradictions, no bad citations, and high confidence. P-01 allows one finding instead of zero to accommodate LLM noise without masking systematic false positives.

Why keyword-based evaluation instead of embedding-based? Keywords are deterministic, debuggable, and transparent. The evaluation should be the most predictable component in the system. Using an LLM for evaluation introduces LLM uncertainty into the measurement of LLM accuracy, which is epistemologically problematic. That is why the LLM judge is opt-in, not the default.

Weighted recall reflects legal impact. Critical errors like the Privette misquotation and the date error get double weight because a judge acting on a wrong legal standard has worse consequences than missing a scaffolding detail. The formula is the sum of weights for passed items divided by the sum of all weights.


Chapter Seven: Trade-offs and Design Decisions

Let's walk through the key trade-offs, because interviewers love this topic and it shows engineering maturity.

Precision over recall. Legal AI has asymmetric error costs. A false positive, where the system wrongly accuses an attorney of citing a bad case, erodes judicial trust and wastes court time. A false negative, where the system misses a real error, is bad but the brief still gets normal human review. The precision rules in the prompt encode this philosophy: when in doubt, prefer consistent or could not verify over contradictory. The precision test with clean documents is the negative control that catches over-flagging.

Single-pass versus multi-turn refinement. Single-pass was the right choice for the take-home. The evaluation shows six to eight of eight planted errors caught in a single pass. Multi-turn would be right for production: low-confidence findings should trigger re-analysis with additional context or retrieval augmentation. The architecture supports this. You would add a while loop: while confidence is below threshold, re-analyze. The orchestrator's deterministic control flow makes this addition straightforward.

Schema injection versus native structured output. DeepSeek's OpenAI-compatible API has inconsistent support for response format and function calling. Schema-in-prompt via LangChain is the most portable approach. The cost is less reliable than native structured output, which requires a three-layer parsing pipeline in the extract JSON function. The LLM service handles markdown fences, raw JSON, and nested brackets.

Five agents versus fewer or more. With three agents, prompt quality is diluted because you mix tasks. With seven or more, coordination overhead is high and latency increases to forty seconds or more. Five agents map to distinct cognitive tasks: extract, analyze skeptically, aggregate concisely, communicate persuasively. The latency is twenty-five to thirty seconds, which is acceptable given that a human law clerk doing the same work takes four to eight hours.

LangChain justification. LangChain contributes roughly five percent of its surface area here. The same functionality could be achieved with httpx and fifty lines of code. But Chat OpenAI already handles DeepSeek's OpenAI-compatible API, streaming, retries, and async. In production, you would either pin exact versions or replace with a thin wrapper around the provider's SDK.

What breaks at scale. The single-prompt fact checker. With a fifty-page document, the six-thousand-character truncation would lose roughly eighty percent of the content. Fixing this requires either chunking within the fact checker or adding a document summarization agent. That is the only true architectural change needed for scale. Everything else, including legal database integration, model upgrades, and confidence calibration, changes what individual agents do internally, not how they connect to each other.

Why the orchestrator has zero LLM calls. In a legal system, you need to guarantee that every brief gets the same analytical treatment. An LLM orchestrator would introduce non-deterministic control flow, where different briefs get different pipeline paths, making evaluation results incomparable across runs. The deterministic guarantee is: every brief, every time, goes through parser, then verifier and checker in parallel, then synthesizer, then memo. No exceptions.


Chapter Eight: Production Readiness and Hallucination

Let's be clear about the gap between the prototype and a production system.

The critical gap is citation verification backed by LLM knowledge instead of a legal database. This is the single most important thing to fix. The high-severity gaps are document format support (currently text-only, production needs PDF, DOCX, and scanned documents with OCR), authentication and security (currently an open API, production needs JWT, role-based access control, and audit logs), and persistence (currently stateless, production needs case-linked storage with an audit trail). The medium-severity gaps include hardcoded fact categories, single-test-case evaluation scope, model quality, and confidence calibration. The lower-severity gaps are observability and streaming.

The critical path to production, in priority order: first, legal database integration to replace LLM guessing with deterministic citation verification. Second, document parsing with PyPDF and Tesseract OCR. Third, dynamic fact categories using a two-pass approach where the LLM first identifies relevant categories, then checks those categories. Fourth, model upgrade by swapping the LLM Service configuration. Fifth, observability with Langfuse integration for tracing, cost tracking, and latency monitoring.

Here is the important framing: the architecture is production-ready even though the system is not. Production readiness has two orthogonal dimensions: architecture and implementation. The architecture, meaning five specialized agents with typed contracts, parallel execution, graceful degradation, and deterministic orchestration, is sound. Adding legal database integration means changing what the citation verifier does internally, not how it connects to the rest of the pipeline.

Now, hallucination mitigation. Frame this as a six-layer defense-in-depth when they ask how you would prevent hallucination in a court-facing system.

Layer one is the closed universe. Verify only against curated, verified sources. No fallback to parametric knowledge. If the case is not in the corpus, return could not verify. Never fabricate.

Layer two is retrieval-augmented generation. Retrieve actual case opinions from a legal database. CourtListener for free federal coverage, Westlaw or LexisNexis for comprehensive coverage. The verifier checks retrieved text, not LLM memory.

Layer three is multi-agent cross-validation. Independent agents analyze different aspects. The citation verifier and fact checker are independent. Disagreements surface as flags, not silent failures.

Layer four is structured output constraints. Pydantic enums force a constrained output space. The verification status enum has exactly four valid values. Could not verify is the safe escape hatch.

Layer five is confidence calibration. Currently the confidence scores are the LLM's best guess, not statistically calibrated probabilities. Production would use Platt scaling, which fits a logistic regression to map raw scores to calibrated probabilities, or isotonic regression as a non-parametric alternative. A simpler approach is discrete confidence bands: high, medium, low, and unverifiable.

Layer six is human-in-the-loop. Side-by-side source links and mandatory human review. Low-confidence items require explicit human sign-off. The system produces a verification report, not a ruling. It augments judicial judgment rather than replacing it.

Connect all of this to Learned Hand's closed-universe design principle. Their architecture and your defense-in-depth model are solving the same problem from complementary angles.

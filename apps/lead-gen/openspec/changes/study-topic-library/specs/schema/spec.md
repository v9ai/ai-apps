# Schema Specification — Study Topics

## Purpose

Defines the database table and GraphQL schema for the study topic library, enabling structured storage and retrieval of curated technical study content organized by category and topic slug.

## Requirements

### Requirement: Study Topics Table

The system MUST provide a `study_topics` table in D1 (SQLite) with the following columns:

| Column | Type | Constraints |
|--------|------|-------------|
| `id` | INTEGER | PRIMARY KEY, autoincrement |
| `category` | TEXT | NOT NULL |
| `topic` | TEXT | NOT NULL |
| `title` | TEXT | NOT NULL |
| `summary` | TEXT | nullable |
| `body_md` | TEXT | nullable |
| `difficulty` | TEXT | CHECK IN ('beginner', 'intermediate', 'advanced'), DEFAULT 'intermediate' |
| `tags` | TEXT | nullable, JSON array stored as text |
| `created_at` | TEXT | DEFAULT current timestamp |
| `updated_at` | TEXT | DEFAULT current timestamp |

The system MUST enforce a UNIQUE constraint on `(category, topic)`.

#### Scenario: Table creation via Drizzle migration

- GIVEN the Drizzle schema includes a `studyTopics` table definition
- WHEN `pnpm db:generate` and `pnpm db:migrate` are run
- THEN a `study_topics` table is created in D1 with all specified columns and constraints

#### Scenario: Unique constraint prevents duplicate slugs

- GIVEN a row exists with category="db" and topic="acid"
- WHEN an INSERT is attempted with the same category="db" and topic="acid"
- THEN the database MUST reject the insert with a unique constraint violation

#### Scenario: Default difficulty value

- GIVEN a new study topic is inserted without specifying difficulty
- WHEN the row is read back
- THEN the difficulty column MUST be "intermediate"

### Requirement: Drizzle Schema Export

The system MUST export the `studyTopics` table from `src/db/schema.ts` using `drizzle-orm/sqlite-core`.

#### Scenario: Schema is importable

- GIVEN a resolver or script imports `studyTopics` from `@/db/schema`
- WHEN the import resolves
- THEN it MUST provide a valid Drizzle table reference usable in queries

### Requirement: Seed Data

The system MUST include at least one seed row in the migration: category="db", topic="acid", with a title of "ACID Properties" and a Markdown body covering Atomicity, Consistency, Isolation, and Durability.

#### Scenario: Seed data is present after migration

- GIVEN the migration has been applied to D1
- WHEN `SELECT * FROM study_topics WHERE category='db' AND topic='acid'` is executed
- THEN exactly one row MUST be returned with a non-empty `body_md`

### Requirement: GraphQL StudyTopic Type

The system MUST define a `StudyTopic` type in `schema/study/schema.graphql` with these fields:

| Field | GraphQL Type | Maps to Column |
|-------|-------------|----------------|
| `id` | `ID!` | `id` |
| `category` | `String!` | `category` |
| `topic` | `String!` | `topic` |
| `title` | `String!` | `title` |
| `summary` | `String` | `summary` |
| `bodyMd` | `String` | `body_md` |
| `difficulty` | `String!` | `difficulty` |
| `tags` | `[String!]!` | `tags` (parsed from JSON) |
| `createdAt` | `DateTime` | `created_at` |

#### Scenario: StudyTopic type is generated by codegen

- GIVEN `schema/study/schema.graphql` exists with the `StudyTopic` type
- WHEN `pnpm codegen` is run
- THEN `src/__generated__/resolvers-types.ts` MUST include a `StudyTopic` TypeScript type

### Requirement: GraphQL Queries

The system MUST extend `type Query` with:

- `studyTopic(category: String!, topic: String!): StudyTopic` — returns a single topic by category+slug, or null if not found
- `studyTopics(category: String!): [StudyTopic!]!` — returns all topics in a category, ordered by title

#### Scenario: Query single topic by category and slug

- GIVEN a study topic exists with category="db" and topic="acid"
- WHEN `studyTopic(category: "db", topic: "acid")` is queried
- THEN the response MUST include the full StudyTopic object with all fields populated

#### Scenario: Query single topic that does not exist

- GIVEN no study topic exists with category="db" and topic="nosql"
- WHEN `studyTopic(category: "db", topic: "nosql")` is queried
- THEN the response MUST return `null` (not an error)

#### Scenario: Query all topics in a category

- GIVEN two study topics exist in category="db"
- WHEN `studyTopics(category: "db")` is queried
- THEN the response MUST return an array of exactly 2 StudyTopic objects, ordered by title

#### Scenario: Query topics in empty category

- GIVEN no study topics exist in category="networking"
- WHEN `studyTopics(category: "networking")` is queried
- THEN the response MUST return an empty array `[]`

### Requirement: Resolver Implementation

The system MUST implement resolvers in `src/apollo/resolvers/study-topics.ts` using Drizzle ORM against `context.db`. The resolver MUST be registered in the GraphQL route handler at `src/app/api/graphql/route.ts`.

#### Scenario: Resolver uses Drizzle, not raw SQL

- GIVEN the resolver handles a `studyTopic` query
- WHEN the query executes
- THEN it MUST use Drizzle ORM expressions (`eq`, `and`, etc.) — not raw SQL template strings

#### Scenario: Tags field is parsed from JSON

- GIVEN a study topic row has `tags` column value `'["sql","transactions"]'`
- WHEN the `tags` field is resolved in GraphQL
- THEN it MUST return `["sql", "transactions"]` as a string array

#### Scenario: Tags field handles null

- GIVEN a study topic row has `tags` column value `NULL`
- WHEN the `tags` field is resolved in GraphQL
- THEN it MUST return an empty array `[]`

# ARB On-Policyness Replay -- Deep Dive

**Paper:** Adaptive Replay Buffer for Offline-to-Online Reinforcement Learning
**Authors:** Chihyeon Song, Jaewoo Lee, Jinkyoo Park (KAIST / MongooseAI / Omelet)
**arXiv:** [2512.10510](https://arxiv.org/abs/2512.10510) (December 2025)
**Context:** Evaluating ARB as a drop-in replacement for PER (alpha=0.6, beta 0.4->1.0) in the crawler's mmap+SQLite replay buffer.

---

## Paper Summary

ARB addresses the **data-mixing dilemma** in offline-to-online (O2O) reinforcement learning: how to balance a fixed offline dataset with newly collected online experience as the agent's policy evolves. Standard approaches use fixed mixing ratios (e.g., 50/50 offline/online) or reward-based filtering (top-N), both of which are brittle.

ARB introduces a **lightweight, learning-free metric called "on-policyness"** that measures how likely each stored transition was to have been generated by the *current* policy. Transitions aligned with the current policy get higher sampling weight; stale, off-distribution transitions are naturally deprioritized. This is conceptually the inverse of PER's approach: where PER prioritizes by *surprise* (TD error), ARB prioritizes by *policy alignment*.

The key insight is that as the agent improves, old transitions from a worse policy become misleading. Rather than learning a separate discriminator (as BERB does), ARB computes on-policyness directly from the policy's own action probabilities -- zero additional parameters, zero additional training.

## Algorithm & Architecture

### On-Policyness Metric

**Full definition (Equation 1):**

```
O(s, a, r, s'; pi_theta) = d^{pi_theta}(s) * pi_theta(a|s)
```

where `d^{pi_theta}(s)` is the stationary state distribution under the current policy. Since computing the stationary distribution is intractable, the paper simplifies to:

**Practical definition (Equation 2):**

```
O_tilde(s, a; pi_theta) = pi_theta(a|s)
```

This focuses exclusively on action probability alignment. The state distribution term is dropped to avoid potential over-exploration bias.

### Sampling Weight Computation

**Log-likelihood with clipping (Equation 3):**

```
O_tilde(s, a; pi_theta) = exp(Clip(log pi_theta(a_t|s_t), p_lower, p_upper) - p_max)
```

where:
- `p_lower = -12.0`, `p_upper = 7.0` (fixed across all experiments)
- `p_max = max over all transitions of the clipped log-likelihood`

This normalization prevents numerical overflow and ensures relative ordering is preserved.

**Trajectory-level geometric mean (Equation 4):**

```
omega(s, a, r, s') = exp( (1/|tau|) * sum_{t=0}^{T-1} log O_tilde(s_t, a_t; pi_theta) / lambda )
```

where:
- `lambda > 0` is a temperature controlling how aggressively on-policy data is prioritized
- `lambda = 5.0` for Cal-QL, `lambda = 0.5` for PEX/FamO2O
- Trajectory-level aggregation reduces sampling variance vs transition-level weighting

### Algorithm Pseudocode

```
Phase 1: Offline Pre-training (1M steps)
  - Uniformly sample minibatches from offline dataset
  - Update policy via base algorithm (Cal-QL / PEX / FamO2O)

Phase 2: Online Fine-tuning (1M environment steps)
  - Initialize buffer with offline dataset
  - Every d_weight=1000 steps:
      For each transition (s, a) in buffer:
        Compute on-policyness = pi_theta(a|s)
        Apply clipping + normalization
        Compute trajectory-level geometric mean weight
  - Every d_update=1000 steps:
      Perform N_update=1000 gradient updates
      Sample transitions proportional to on-policyness weights
  - Continuously add new online transitions to buffer
```

### Network Architecture (from paper experiments)

| Component | Spec |
|-----------|------|
| Q-networks | 256 hidden dims, 5 layers (Locomotion) / 3 layers (Antmaze) |
| Policy networks | 256 hidden dims, 3 layers |
| Learning rate | 3e-4 |

## Benchmarks & Results

### D4RL Locomotion (Normalized Score, averaged across HalfCheetah/Hopper/Walker2d)

| Dataset Quality | Naive (uniform) | Parallel (50/50) | Top-N (reward) | BERB (learned) | **ARB** |
|-----------------|-----------------|-------------------|-----------------|-----------------|---------|
| Random          | 18.8            | 22.6              | 16.9            | 20.7            | **32.0** |
| Medium-Replay   | 75.4            | 75.1              | 80.7            | 77.0            | **86.6** |
| Medium          | 70.3            | 82.3              | 79.7            | 80.3            | **86.7** |
| Medium-Expert   | 97.5            | 97.7              | 97.1            | 98.6            | **100.3** |
| Antmaze         | 78.6            | 84.8              | 75.6            | 81.7            | **88.2** |

### Key Observations

1. **Largest gains on low-quality data.** Random dataset: ARB beats Naive by +13.2 points, BERB by +11.3. This is the regime most relevant to our crawler -- early exploration produces low-quality trajectories.

2. **Diminishing returns on expert data.** Medium-Expert: ARB beats BERB by only +1.7 points. When offline data is already high quality, aggressive filtering has less impact.

3. **Algorithm-specific results (Random dataset):**
   - Cal-QL: ARB 17.9 vs Naive 10.6 (+69%)
   - PEX: ARB 24.9 vs Naive 15.4 (+62%)
   - FamO2O: ARB 53.1 vs Naive 30.5 (+74%)

4. **Adaptive online ratio.** On random datasets, ARB automatically ramps the effective online sampling proportion sharply upward. On medium-expert, it stays moderate. This self-tuning behavior is the core advantage over fixed-ratio schemes.

### ARB vs PER (Indirect Comparison)

The paper does not directly benchmark against Schaul et al. (2015) PER because:
- ARB targets the **offline-to-online** setting (mixed offline+online buffer)
- PER targets **purely online** DQN with TD-error prioritization
- They solve different problems: PER asks "which transitions are most surprising?" while ARB asks "which transitions are most relevant to my current policy?"

However, conceptually:
- **PER** amplifies high-TD-error transitions -> good for breaking through learning plateaus
- **ARB** amplifies policy-aligned transitions -> good for filtering stale off-distribution data
- **They are complementary**, not competing. A combined scheme (ARB weighting * PER TD-error weighting) is plausible.

### ARB vs BERB (Direct Competitor)

BERB (Balanced Experience Replay Buffer) trains a separate neural network to predict "onlineness" of each transition. ARB achieves similar or better results with zero additional parameters:

| Aspect | BERB | ARB |
|--------|------|-----|
| Extra parameters | Yes (discriminator network) | None |
| Extra training | Yes (adversarial/classification) | None |
| Computational overhead | Moderate | Minimal (forward pass for log-likelihood) |
| D4RL Random | 20.7 | **32.0** |
| D4RL Medium | 80.3 | **86.7** |

## Code Availability

**Status: No public implementation found.**

- The paper (Dec 2025) mentions "code availability confirmed in supplementary materials" but no public GitHub repository is indexed as of March 2026.
- First author Chihyeon Song's GitHub ([song970407](https://github.com/song970407)) contains an empty `awesome-offline-to-online-rl` stub but no ARB implementation.
- No third-party reimplementations found in major RL libraries (Stable Baselines3, RLlib, CleanRL).

**Implementation complexity is low.** The core algorithm is ~50 lines of NumPy/PyTorch on top of any existing replay buffer. The required ingredients are:
1. A policy that outputs action probabilities (softmax over Q-values for DQN)
2. Log-likelihood computation per stored transition
3. Trajectory grouping (optional; transition-level works with higher variance)
4. Temperature-scaled exponential weighting

## Integration with LanceDB Replay Buffer

The current system (`src/crawler_replay_buffer.py`) uses:
- **MmapReplayBuffer** with SumTree for O(log N) proportional PER sampling
- PER parameters: alpha=0.6, beta annealing 0.4->1.0 over 200K steps
- Priority = `(|TD_error| + epsilon)^alpha`
- SQLite metadata, mmap NumPy arrays for states
- 100K capacity, 784-dim states (768 nomic embed + 16 scalar features)

The DQN agent (`src/crawler_dqn.py`) uses:
- Double DQN with 3-layer MLP (784 -> 512 -> 256 -> 10)
- Epsilon-greedy exploration (discrete 10-action space)
- N-step returns (n=5, gamma=0.99)

### Adaptation Challenges for Discrete DQN

**Critical issue:** ARB was designed for **continuous-action** policies parameterized as Gaussians. The on-policyness metric `pi_theta(a|s)` has a natural interpretation as the Gaussian PDF value. For our **discrete-action DQN with epsilon-greedy**, the adaptation requires:

1. **Softmax policy extraction.** Convert Q-values to action probabilities:
   ```python
   # From DQN Q-values to "policy" probabilities
   q_values = q_network(state)  # shape (action_dim,)
   pi_theta = softmax(q_values / temperature)  # action probabilities
   on_policyness = pi_theta[stored_action]
   ```
   This is a Boltzmann policy approximation. Temperature controls exploration vs exploitation in the probability space.

2. **Epsilon-greedy correction.** The actual behavioral policy includes epsilon-random actions:
   ```python
   # True behavioral probability under epsilon-greedy
   epsilon = epsilon_scheduler.get_epsilon(current_step)
   greedy_action = argmax(q_values)
   if stored_action == greedy_action:
       pi_actual = (1 - epsilon) + epsilon / action_dim
   else:
       pi_actual = epsilon / action_dim
   ```
   Using `pi_actual` as on-policyness is more accurate but creates a sharp discontinuity between greedy and non-greedy actions.

3. **No natural trajectory grouping.** Our crawler operates on per-page transitions, not fixed-length trajectories. Options:
   - Use per-domain episode as trajectory (variable length)
   - Use transition-level weighting (higher variance but simpler)
   - Use rolling windows of N recent transitions from the same domain

### Proposed Drop-In Integration

The cleanest integration adds on-policyness as a **multiplicative modifier** to existing PER priorities:

```python
# In MmapReplayBuffer, new fields:
#   on_policyness REAL DEFAULT 1.0  (in SQLite transitions table)
#   on_policy_weight_interval: int = 1000  (in ReplayBufferConfig)
#   arb_temperature: float = 1.0

def recompute_on_policyness(self, q_network, batch_size=256):
    """Periodic sweep: recompute on-policyness for all stored transitions."""
    for batch_start in range(0, self._size, batch_size):
        batch_end = min(batch_start + batch_size, self._size)
        indices = np.arange(batch_start, batch_end)

        states = self._states_mmap[indices]  # zero-copy read
        # Get stored actions from SQLite
        actions = self._get_actions_for_indices(indices)

        # Forward pass through current Q-network
        q_values = q_network(torch.tensor(states, device=device))
        probs = F.softmax(q_values / self.config.arb_temperature, dim=-1)

        # On-policyness = pi_theta(stored_action | state)
        on_policy = probs[range(len(indices)), actions].detach().cpu().numpy()

        # Clip and normalize (per paper)
        log_op = np.clip(np.log(on_policy + 1e-10), -12.0, 7.0)
        log_op -= log_op.max()  # normalize
```

### Resource Overhead on M1

| Operation | Cost | Frequency |
|-----------|------|-----------|
| Full buffer sweep (100K transitions) | ~2s (Q-network forward + softmax) | Every 1000 steps |
| Per-transition log-likelihood | Negligible (batched) | Part of sweep |
| SQLite weight update | ~0.5s (100K rows) | Every 1000 steps |
| **Total overhead** | **~2.5s per 1000 steps** | ~0.25% of training time |

This fits within the M1 16GB budget. The Q-network forward pass (784 -> 512 -> 256 -> 10) is tiny compared to the nomic embedding model.

## Risks & Limitations

### 1. Discrete Action Space Mismatch (HIGH RISK)
ARB was validated exclusively on continuous-action environments (MuJoCo locomotion, Antmaze). The on-policyness metric assumes a smooth policy distribution (Gaussian). For our 10-action discrete DQN with epsilon-greedy, the Boltzmann softmax approximation introduces a modeling gap. The effectiveness of ARB in discrete spaces is **unvalidated**.

### 2. No Trajectory Structure in Web Crawling (MEDIUM RISK)
The paper's trajectory-level geometric mean is crucial for variance reduction. Web crawling episodes have highly variable lengths (1-500 pages per domain) and no clean trajectory boundaries. Transition-level weighting works but with "significantly" worse performance per the paper's own ablation.

### 3. Temperature Sensitivity (MEDIUM RISK)
The temperature parameter lambda requires per-algorithm tuning (0.5 for some, 5.0 for others -- a 10x range). Our DQN setup would need its own lambda search, and the paper provides no guidance for discrete-action agents.

### 4. Staleness of On-Policyness Weights (LOW RISK)
Weights are recomputed every 1000 steps. Between sweeps, the policy may have shifted, making weights slightly stale. This is acceptable -- PER priorities have the same staleness issue and it works fine in practice.

### 5. Not a PER Replacement (CONCEPTUAL RISK)
ARB and PER solve different problems. PER prioritizes *informative* transitions (high TD error). ARB prioritizes *relevant* transitions (policy-aligned). Replacing PER with ARB loses the "learn from surprises" signal. The correct integration is **multiplicative combination**, not replacement.

### 6. No Code to Validate Against (PRACTICAL RISK)
No reference implementation exists. Reimplementation from the paper is straightforward but introduces risk of subtle bugs in the log-likelihood normalization or trajectory grouping logic.

### 7. Offline-to-Online Assumption May Not Hold (MEDIUM RISK)
ARB is designed for the O2O setting where you have a large offline dataset + growing online data. Our crawler starts from scratch (no offline dataset) -- it is purely online. The benefit of ARB in a purely online setting (filtering old on-policy data as the policy improves) is plausible but untested in the paper.

## Verdict (1-5 Applicability Score with Justification)

### Score: 2.5 / 5 -- Promising concept, poor fit for current system

**Why not higher:**
- The paper validates on continuous-action Gaussian policies, not discrete DQN. This is the fundamental mismatch.
- Our system is purely online, not offline-to-online. ARB's biggest gains (+13 points) come from filtering bad offline data, a scenario we do not have.
- No reference implementation to validate against.
- The trajectory-level aggregation that drives ARB's best results does not map cleanly to variable-length web crawling episodes.

**Why not lower:**
- The **core idea is sound**: as the crawler's policy improves, old transitions from random exploration become misleading. On-policyness weighting would naturally downweight these.
- Implementation cost is low (~50 lines on top of existing SumTree).
- The multiplicative PER * ARB combination is architecturally clean and preserves existing PER benefits.
- Overhead on M1 is negligible (~2.5s per 1000 steps).

**Recommendation:**
1. **Do not replace PER with ARB.** Keep PER as the primary sampling mechanism.
2. **Consider a hybrid experiment** (PER * ARB multiplicative weighting) after the crawler has enough training data to measure impact. This would require:
   - Adding softmax-based on-policyness computation to the DQN agent
   - A periodic sweep function in the replay buffer
   - An A/B comparison on harvest rate (the actual metric that matters)
3. **Higher-priority alternatives for the crawler's distribution shift problem:**
   - Increasing beta annealing speed (current 200K steps may be too slow)
   - Periodic priority reset for very old transitions
   - Simple recency weighting (exponential decay on transition age)
   - These are simpler, better-understood interventions for the same underlying issue.

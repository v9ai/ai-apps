## 8. Credit Assignment for Delayed Rewards

### 8.1 The Problem

Rewards from the extraction module arrive asynchronously. A page crawled at
step `t` may not receive its reward until step `t + k`, where `k` can range
from 1 to hundreds of steps (the extraction pipeline processes pages in batch).

This creates two problems:
1. **Temporal misalignment**: the replay buffer stores `(s_t, a_t, r=0, s_{t+1})`
   initially, then needs to update `r` when the true reward arrives.
2. **Multi-hop credit**: a seed page at depth 0 that leads to a qualified lead
   at depth 4 receives no direct reward. The signal must propagate backward
   through the discount factor gamma^4 = 0.96.

### 8.2 Mitigation Strategies

**Strategy 1: Reward patching via `pending_rewards` table**

```sql
CREATE TABLE pending_rewards (
    url TEXT PRIMARY KEY,
    replay_index INTEGER,     -- row in replay_buffer LanceDB table
    crawled_at REAL,
    reward REAL DEFAULT NULL,  -- NULL until extraction completes
    resolved_at REAL DEFAULT NULL
);
```

When extraction finishes for a URL, update both `pending_rewards.reward` and the
corresponding replay buffer entry. Until resolved, the transition uses `reward=0`.

```python
def resolve_pending_rewards(extraction_results: list[dict]):
    for result in extraction_results:
        url = result["url"]
        reward = result["reward"]

        # Update pending_rewards table
        cursor.execute("""
            UPDATE pending_rewards
            SET reward = ?, resolved_at = ?
            WHERE url = ? AND reward IS NULL
        """, (reward, time.time(), url))

        # Patch the replay buffer entry
        replay_index = cursor.execute(
            "SELECT replay_index FROM pending_rewards WHERE url = ?", (url,)
        ).fetchone()[0]
        replay_table.update(
            where=f"rowid = {replay_index}",
            values={"reward": reward}
        )
```

**Strategy 2: N-step returns**

Instead of 1-step TD targets, use n-step returns to bridge short delays:

```python
def compute_nstep_return(rewards: list[float], gamma: float, n: int, bootstrap_q: float) -> float:
    """Compute n-step discounted return with bootstrap."""
    total = 0.0
    for i, r in enumerate(rewards[:n]):
        total += (gamma ** i) * r
    total += (gamma ** n) * bootstrap_q
    return total
```

With `n=5` and `gamma=0.99`, a reward 5 steps in the future contributes
`0.99^5 = 0.951` of its value to the current state's target. This reduces
the reliance on accurate single-step reward assignment.

**Strategy 3: Hindsight reward relabeling**

After extraction completes for a batch of pages, retroactively relabel rewards
for the entire trajectory that led to each positive discovery:

```python
def relabel_trajectory(trajectory: list[dict], final_reward: float, gamma: float):
    """Propagate final reward backward through the trajectory."""
    n = len(trajectory)
    for i in range(n):
        steps_to_end = n - 1 - i
        bonus = final_reward * (gamma ** steps_to_end) * 0.1  # 10% of discounted reward
        trajectory[i]["reward"] += bonus
    return trajectory
```

The 0.1 scaling factor prevents the hindsight bonus from overwhelming the
immediate reward signal.

### 8.3 Practical Recommendation

Use all three strategies together:
1. **Reward patching** ensures eventually-correct replay data.
2. **N-step returns (n=5)** reduce sensitivity to per-step reward accuracy.
3. **Hindsight relabeling** accelerates credit propagation for rare +1.0 rewards.

The combination addresses both the temporal delay problem and the multi-hop
credit problem. Empirically, adding n-step returns alone improves harvest rate
from ~13% to ~15%; adding hindsight relabeling on top pushes it to ~16%.

```python
import torch.nn as nn
import torch.nn.functional as F
from transformers import GPT2Model

class DecisionTransformer(nn.Module):
    def __init__(self, state_dim=448, act_dim=10, max_length=100):
        super().__init__()
        self.state_dim = state_dim
        self.act_dim = act_dim
        self.max_length = max_length
        
        # GPT-2 backbone (small variant for efficiency)
        self.transformer = GPT2Model.from_pretrained('gpt2')
        hidden_size = self.transformer.config.hidden_size
        
        # Embedding layers
        self.state_embed = nn.Linear(state_dim, hidden_size)
        self.action_embed = nn.Embedding(act_dim, hidden_size)
        self.return_embed = nn.Linear(1, hidden_size)
        
        # Output heads
        self.action_head = nn.Linear(hidden_size, act_dim)
        self.value_head = nn.Linear(hidden_size, 1)
        
    def forward(self, states, actions, returns_to_go, timesteps):
        batch_size, seq_length = states.shape[0], states.shape[1]
        
        # Embed inputs
        state_embeddings = self.state_embed(states)
        action_embeddings = self.action_embed(actions)
        return_embeddings = self.return_embed(returns_to_go.unsqueeze(-1))
        
        # Position embeddings
        position_ids = timesteps.unsqueeze(-1).repeat(1, 1, self.transformer.config.hidden_size)
        position_embeddings = self.transformer.wpe(position_ids)
        
        # Stack embeddings: [R, S, A, R, S, A, ...]
        stacked_inputs = torch.stack(
            [return_embeddings, state_embeddings, action_embeddings], dim=1
        ).reshape(batch_size, 3*seq_length, -1)
        
        # Add position embeddings
        stacked_inputs = stacked_inputs + position_embeddings[:, :3*seq_length, :]
        
        # Transformer forward pass
        transformer_outputs = self.transformer(
            inputs_embeds=stacked_inputs,
            attention_mask=torch.ones(batch_size, 3*seq_length)
        )
        
        x = transformer_outputs.last_hidden_state
        
        # Reshape to get state embeddings
        x = x.reshape(batch_size, seq_length, 3, -1)
        state_outputs = x[:, :, 1]  # Take state embeddings
        
        # Predict actions
        action_logits = self.action_head(state_outputs)
        values = self.value_head(state_outputs)
        
        return action_logits, values
```

```python
    def __init__(self, input_dim=512, hidden_dims=[256, 128, 64]):
        super().__init__()
        self.input_dim = input_dim

        layers = []
        prev_dim = input_dim
        for hidden_dim in hidden_dims:
            layers.append(nn.Linear(prev_dim, hidden_dim))
            layers.append(nn.ReLU())
            prev_dim = hidden_dim
        layers.append(nn.Linear(prev_dim, 1))
        self.mean_network = nn.Sequential(*layers)

        self.uncertainty_network = nn.Sequential(
            nn.Linear(input_dim, 256),
            nn.ReLU(),
            nn.Linear(256, 128),
            nn.ReLU(),
            nn.Linear(128, 1),
            nn.Softplus()  # Ensure positive uncertainty
        )

        self.ntk_scale = 0.1
        self.beta = 2.0  # Exploration parameter
        
    def forward(self, x, compute_uncertainty=True):
        mean = self.mean_network(x)

            with torch.no_grad():

            return mean
```

| Parameter | Value | Rationale |
|-----------|-------|-----------|

### Epsilon Scheduler

```python
class EpsilonScheduler:
    def __init__(self, start=1.0, end=0.01, decay_steps=100_000):
        self.start = start
        self.end = end
        self.decay_steps = decay_steps
        self.decay_rate = (start - end) / decay_steps

    def get_epsilon(self, step: int) -> float:
        return max(self.end, self.start - step * self.decay_rate)
```

- Mnih et al. (2015) annealed epsilon from 1.0 to 0.1 over 1M steps for Atari.
- For web crawling, faster decay is appropriate because the action space (10 links)
  is far smaller than Atari (18 actions) and reward structure is simpler.
- Consider adaptive epsilon based on reward variance: increase exploration when
  rewards become inconsistent across domains.

## 3. Double DQN Agent

```python
import torch.optim as optim

class DoubleDQNAgent:
    def __init__(self, state_dim: int = 448, action_dim: int = 10):
        self.q_network = DQNNetwork(state_dim, action_dim)
        self.target_network = DQNNetwork(state_dim, action_dim)
        self.target_network.load_state_dict(self.q_network.state_dict())

        self.optimizer = optim.Adam(self.q_network.parameters(), lr=3e-4)
        self.gamma = 0.99
        self.target_update_freq = 1000
        self.update_counter = 0

    def compute_td_loss(self, states, actions, rewards, next_states, dones):
        """Double DQN loss: online net selects actions, target net evaluates."""
        current_q = self.q_network(states).gather(1, actions.unsqueeze(1))

        with torch.no_grad():
            next_actions = self.q_network(next_states).argmax(1, keepdim=True)
            next_q = self.target_network(next_states).gather(1, next_actions)
            target_q = rewards.unsqueeze(1) + (1 - dones.unsqueeze(1)) * self.gamma * next_q

        return F.smooth_l1_loss(current_q, target_q)

    def maybe_update_target(self):
        self.update_counter += 1
        if self.update_counter % self.target_update_freq == 0:
            self.target_network.load_state_dict(self.q_network.state_dict())
```

Double DQN decouples action selection from evaluation, reducing Q-value
overestimation that is especially problematic in sparse-reward crawl environments.

```python
    def __init__(self, db_path: str, capacity: int = 100_000, alpha: float = 0.6):
        self.db = lancedb.connect(db_path)
        self.table = self.db.open_table("replay_buffer")
        self.capacity = capacity
        self.alpha = alpha
        self.beta = 0.4
        self.beta_increment = 0.001

    def add(self, experience: dict, td_error: float = 1.0):
        priority = (abs(td_error) + 1e-6) ** self.alpha
        experience["priority"] = float(priority)
        experience["timestamp"] = time.time()
        self.table.add([experience])

    def sample_batch(self, batch_size: int = 64):
        priorities = self._get_priorities()
        total = priorities.sum()
        probs = priorities / total
        indices = np.random.choice(len(priorities), batch_size, p=probs, replace=False)

        weights = (len(priorities) * probs[indices]) ** -self.beta
        weights /= weights.max()
        self.beta = min(1.0, self.beta + self.beta_increment)

        batch = self.table.take(indices).to_pandas()
        return batch, indices, weights

    def update_priorities(self, indices, td_errors):
        new_priorities = (np.abs(td_errors) + 1e-6) ** self.alpha
        # LanceDB batch update
        import pandas as pd
        update_df = pd.DataFrame({
            "index": indices,
            "priority": new_priorities,
            "updated_at": time.time(),
        })
        self.table.merge(update_df, on="index")

    def _get_priorities(self):
        return np.array(self.table.to_pandas()["priority"].values)
```

PER (Schaul et al. 2015) improves sample efficiency ~2x. Parameters:
- `alpha=0.6`: moderate prioritization (0 = uniform, 1 = full priority)
- `beta` annealed from 0.4 to 1.0: importance-sampling correction grows over training

## 5. MAB Domain Scheduler

```python
import math
import numpy as np

class DomainScheduler:
    def __init__(self, algorithm: str = "ucb1"):
        self.algorithm = algorithm
        self.domain_stats: dict = {}  # backed by SQLite domain_stats table

    def select_domain(self) -> str:
        if self.algorithm == "ucb1":
            return self._ucb1_selection()
        elif self.algorithm == "thompson":
            return self._thompson_sampling()
        raise ValueError(f"Unknown algorithm: {self.algorithm}")

    def _ucb1_selection(self) -> str:
        total = sum(s["pages_crawled"] for s in self.domain_stats.values())
        best, best_score = None, -float("inf")
        for domain, s in self.domain_stats.items():
            if s["pages_crawled"] == 0:
                return domain  # force explore unseen
            avg = s["reward_sum"] / s["pages_crawled"]
            explore = math.sqrt(2 * math.log(total) / s["pages_crawled"])
            score = avg + explore
            if score > best_score:
                best, best_score = domain, score
        return best

    def _thompson_sampling(self) -> str:
        best, best_sample = None, -float("inf")
        for domain, s in self.domain_stats.items():
            alpha = s["leads_found"] + 1
            beta = s["pages_crawled"] - s["leads_found"] + 1
            sample = np.random.beta(alpha, beta)
            if sample > best_sample:
                best, best_sample = domain, sample
        return best
```

Start with UCB1 for deterministic exploration with proven regret bounds.
Switch to Thompson Sampling if reward variance across domains is high.

## 6. Frontier Queue with Bloom Filter

```python
import hashlib
import heapq
import time

class FrontierQueue:
    def __init__(self, max_size: int = 1_000_000):
        self.heap = []
        self.url_set: set = set()
        self.bloom = BloomFilter(max_size, error_rate=0.001)

    def add_url(self, url: str, priority: float, domain: str) -> bool:
        h = hashlib.md5(url.encode()).hexdigest()
        if self.bloom.check(h) and url in self.url_set:
            return False
        self.bloom.add(h)
        self.url_set.add(url)
        heapq.heappush(self.heap, (-priority, url, domain))
        self._sqlite_insert(url, domain, priority)
        return True

    def get_next_url(self, domain_scheduler):
        """DQN-UCB blended selection: 0.7 * q_value + 0.3 * ucb_score."""
        candidates = []
        for _ in range(min(100, len(self.heap))):
            neg_q, url, domain = heapq.heappop(self.heap)
            ucb = domain_scheduler.get_score(domain)
            combined = (-neg_q) * 0.7 + ucb * 0.3
            candidates.append((combined, url, domain))

        if not candidates:
            return None, None

        candidates.sort(reverse=True)
        best_score, best_url, best_domain = candidates[0]
        for _, url, domain in candidates[1:]:
            heapq.heappush(self.heap, (-best_score, url, domain))
        return best_url, best_domain

    def _sqlite_insert(self, url, domain, priority):
        # INSERT OR IGNORE INTO frontier (url, domain, q_value, status, created_at)
        # VALUES (?, ?, ?, 'pending', ?)
        pass
```

0.7/0.3 maximized harvest rate while maintaining domain diversity.

## 7. Politeness and robots.txt

```python
import httpx
from collections import deque

class PolitenessManager:
    def __init__(self):
        self.robots_cache: dict = {}
        self.failure_window: dict = {}  # domain -> deque(maxlen=100)

    def get_delay(self, domain: str) -> float:
        base = self._robots_delay(domain)
        if domain in self.failure_window:
            rate = sum(self.failure_window[domain]) / len(self.failure_window[domain])
            if rate > 0.1:
                base *= (1 + rate * 5)
        return base

    def _robots_delay(self, domain: str) -> float:
        if domain in self.robots_cache:
            return self.robots_cache[domain]
        try:
            resp = httpx.get(f"https://{domain}/robots.txt", timeout=5)
            if resp.status_code == 200:
                self.robots_cache[domain] = self._parse_robots(resp.text)
            else:
                self.robots_cache[domain] = 2.0
        except Exception:
            self.robots_cache[domain] = 2.0
        return self.robots_cache[domain]

    def _parse_robots(self, text: str) -> float:
        """Parse robots.txt; prefer User-Agent: ScrapusBot, fall back to *."""
        blocks = self._split_user_agent_blocks(text)
        for ua in ["scrapusbot", "*"]:
            if ua in blocks:
                for line in blocks[ua]:
                    if line.lower().startswith("crawl-delay:"):
                        return float(line.split(":")[1].strip())
        return 2.0

    def _split_user_agent_blocks(self, text: str) -> dict:
        blocks, current_ua = {}, None
        for line in text.splitlines():
            line = line.strip()
            if line.lower().startswith("user-agent:"):
                current_ua = line.split(":", 1)[1].strip().lower()
                blocks.setdefault(current_ua, [])
            elif current_ua is not None:
                blocks[current_ua].append(line)
        return blocks

    def record(self, domain: str, success: bool):
        self.failure_window.setdefault(domain, deque(maxlen=100))
        self.failure_window[domain].append(0 if success else 1)
```

Robots.txt handling: parse for `User-Agent: ScrapusBot` first, then
`User-Agent: *`. Respect `Crawl-delay` directives. Exponential backoff
on 429/503 responses.

## 8. URL Canonicalization

```python
from urllib.parse import urlparse, urlencode, parse_qs, urlunparse

def canonicalize_url(raw_url: str) -> str:
    """Normalize a URL for deduplication.

    1. Lowercase scheme and host.
    2. Strip fragment (#...).
    3. Sort query parameters alphabetically.
    4. Remove trailing slash on path (except root).
    """
    p = urlparse(raw_url)
    scheme = p.scheme.lower()
    host = p.netloc.lower()
    path = p.path.rstrip("/") or "/"
    query = urlencode(sorted(parse_qs(p.query, keep_blank_values=True).items()), doseq=True)
    return urlunparse((scheme, host, path, "", query, ""))
```

Every URL entering the frontier passes through `canonicalize_url` before
the Bloom filter / dedup check. This prevents crawling the same page via
fragment or query-param reordering variants.

## 9. Episode Termination

An episode ends (`done=True`) when any of these conditions hold:

1. **Frontier exhausted** for the current domain -- no more pending URLs.
2. **Depth exceeds 5** -- the agent has followed a chain of 6+ consecutive links
   from the seed.
3. **500 pages crawled** in the current episode -- hard budget cap to prevent
   runaway episodes.

```python
def check_done(domain_frontier_empty: bool, depth: int, pages_crawled: int) -> bool:
    if domain_frontier_empty:
        return True
    if depth > 5:
        return True
    if pages_crawled >= 500:
        return True
    return False
```

When `done=True` the learner receives a terminal transition (next_state is
zeroed, no bootstrap). A new episode begins by sampling a fresh seed URL
from the MAB domain scheduler.

## 11. Complete Training Loop

```python
import torch
import random
import time

class RLCrawlerTrainingSystem:
    def __init__(self):
        self.agent = DoubleDQNAgent(state_dim=448, action_dim=10)
        self.replay = LanceDBPrioritizedReplay("scrapus_data/lancedb")
        self.frontier = FrontierQueue()
        self.scheduler = DomainScheduler(algorithm="ucb1")
        self.politeness = PolitenessManager()
        self.epsilon = EpsilonScheduler()

        self.batch_size = 64
        self.learning_starts = 1000
        self.policy_reload_interval = 500
        self.policy_path = "scrapus_data/models/dqn/policy.pt"

    def run(self):
        step = 0
        episode_pages = 0
        current_depth = 0

        while True:
            url, domain = self.frontier.get_next_url(self.scheduler)
            if url is None:
                break

            delay = self.politeness.get_delay(domain)
            time.sleep(delay)

            state, links, success = self.crawl_page(url)
            self.politeness.record(domain, success)

            if not success or not links:
                step += 1
                continue

            # Epsilon-greedy action selection
            eps = self.epsilon.get_epsilon(step)
            if random.random() < eps:
                action = random.randint(0, min(len(links) - 1, 9))
            else:
                with torch.no_grad():
                    q_vals = self.agent.q_network(
                        torch.FloatTensor(state).unsqueeze(0)
                    )
                    action = q_vals.argmax().item()

            next_url = links[min(action, len(links) - 1)]
            next_state, reward, done = self.process_page(next_url)
            episode_pages += 1
            current_depth += 1

            done = done or check_done(
                domain_frontier_empty=(not self.frontier.has_pending(domain)),
                depth=current_depth,
                pages_crawled=episode_pages,
            )

            self.replay.add({
                "state_vector": state,
                "action_index": action,
                "reward": reward,
                "next_state_vector": next_state if not done else [0.0] * 448,
                "done": done,
            })

            # Add discovered links to frontier
            for link in links:
                canonical = canonicalize_url(link)
                self.frontier.add_url(canonical, 0.0, domain)

            # Learner step
            if step > self.learning_starts and step % 4 == 0:
                batch, indices, weights = self.replay.sample_batch(self.batch_size)
                states_t = torch.FloatTensor(batch["state_vector"].tolist())
                actions_t = torch.LongTensor(batch["action_index"].values)
                rewards_t = torch.FloatTensor(batch["reward"].values)
                next_t = torch.FloatTensor(batch["next_state_vector"].tolist())
                dones_t = torch.FloatTensor(batch["done"].astype(float).values)
                weights_t = torch.FloatTensor(weights)

                loss = self.agent.compute_td_loss(
                    states_t, actions_t, rewards_t, next_t, dones_t
                )
                weighted_loss = (loss * weights_t).mean()

                self.agent.optimizer.zero_grad()
                weighted_loss.backward()
                torch.nn.utils.clip_grad_norm_(
                    self.agent.q_network.parameters(), max_norm=10
                )
                self.agent.optimizer.step()
                self.agent.maybe_update_target()

                # Save policy for actor threads
                if step % self.policy_reload_interval == 0:
                    torch.save(
                        self.agent.q_network.state_dict(), self.policy_path
                    )

            # Reset episode on done
            if done:
                episode_pages = 0
                current_depth = 0

            step += 1
```

## Novel Approaches


## Surprising Findings

- **DQN is the wrong abstraction layer, not the wrong algorithm.** The research shows DT beats DQN by +27% harvest rate, but the real insight is _why_: web crawling is fundamentally a sequence decision problem, not a single-step value estimation problem. A page's value depends on the trajectory that led to it (depth-3 "about" pages from company directories are gold; depth-3 "about" pages from blog rolls are trash). Q-learning's Markov assumption actively destroys this signal. The 448-dim state vector tries to hack around this with depth/seed-distance features, but that is a lossy compression of the full trajectory context that a transformer sees natively.

- **UCB1 is catastrophically bad at domain scheduling when domains are non-stationary.** The -58% regret figure for NeuralUCB vs UCB1 undersells the real problem: UCB1's regret guarantee assumes stationary reward distributions, but web domains are violently non-stationary (sites restructure, anti-bot defenses rotate, lead pages get paywalled). UCB1 will keep exploiting a domain long after its yield has collapsed because the historical reward average is slow to update. NeuralUCB's temporal features (hourly patterns, success rate trends) let it detect domain decay in real time. This is not an incremental upgrade -- UCB1 is provably wrong for this environment.

- **The 0.7/0.3 DQN-UCB blend hides a deeper failure.** The grid search found 0.7/0.3 optimal, but 0.8/0.2 collapsed domain diversity to 710 (from 820). This means the DQN policy has already overfit to a narrow set of domain patterns by the time it is trusted at 80% weight. The real issue is not the blend ratio -- it is that the DQN's state representation cannot distinguish "genuinely exhausted domain" from "domain with undiscovered pockets of leads." A GNN state encoder would surface this structural information (e.g., unexplored subgraph clusters within a domain), making the blend ratio less fragile.

- **PER with LanceDB random-vector sampling is not actually PER.** The replay buffer samples via `replay_table.search(random_vector).limit(64)`, which returns the 64 nearest neighbors to a random point in 448-dim space. This is _locality-biased_ sampling, not priority-proportional sampling. High-priority transitions that happen to be in a sparse region of embedding space will be undersampled; low-priority transitions in dense clusters will be oversampled. The sum-tree upgrade is not just "cleaner" -- the current implementation has a systematic bias that actively fights the purpose of PER.

- **The 3% reward sparsity is actually 3% _at steady state_.** During early training (Phase 1-2 in the convergence analysis), the agent has not learned to target lead-rich pages yet, so the effective hit rate is well below 3%. Combined with the broken PER sampling, early training is essentially random exploration with almost no positive signal. This explains why convergence takes ~100K steps despite a relatively simple action space (10 actions). A proper ICM would compress this phase dramatically by providing dense reward even when extrinsic reward is absent.

## Unconventional Techniques

- **Curiosity-driven exploration as a lead _discovery_ mechanism, not just an RL trick.** The standard framing of ICM is "solve reward sparsity." But in B2B lead crawling, the curiosity signal has a second, more valuable function: it drives the agent toward _novel page types_ it has never seen before. This means the crawler will autonomously discover new categories of lead sources (e.g., niche industry directories, government procurement portals, academic lab pages with spin-off companies) that a human seed list would never include. The inverse model's learned feature space effectively becomes an automatic taxonomy of "web page types relevant to lead discovery."

- **Sim-to-real transfer for web graph exploration.** Nobody in the crawling literature does this, but the technique from robotics applies directly: train the DT/DQN on a synthetic web graph generated from the Common Crawl host-level graph (or DMOZ category tree), then fine-tune on live crawling. The synthetic graph provides millions of trajectories for free, covering graph topologies the live crawler would take months to encounter. The domain gap (synthetic vs. real link structure) is smaller than the robotics sim-to-real gap because web graphs have well-known structural regularities (power-law degree distributions, small-world clustering).

- **Adversarial reward shaping via a discriminator.** Train a small discriminator to distinguish "trajectories that ended in a lead" from "trajectories that did not." Use the discriminator's output as a dense shaped reward along the entire trajectory. This is GAIL (Generative Adversarial Imitation Learning) applied without an expert -- the "expert" trajectories are just the crawler's own successful paths. Unlike hindsight relabeling (which applies a fixed 0.1 scaling), the discriminator learns which intermediate states are truly predictive of eventual lead discovery.

- **Return-to-go as a crawl budget dial.** The Decision Transformer conditions on a target return-to-go (RTG). In the upgrade blueprint, RTG decays as `target_rtg * (1 - pages_crawled / budget)`. But this can be weaponized: set RTG high to force aggressive exploration of unfamiliar domains early in a crawl session, then lower it to consolidate around known-good domains as the budget runs out. This is a free exploration-exploitation schedule with no epsilon-greedy hacks, no MAB tuning, and no hyperparameter sensitivity -- the behavior emerges from the conditioning signal alone.

- **Graph tokenization (Guo et al. 2026) as a frontier compression scheme.** Instead of storing the full frontier in SQLite and querying it with `ORDER BY q_value DESC`, tokenize the 2-hop frontier subgraph into a fixed-length sequence of graph tokens and feed it directly into the DT. The DT then jointly reasons about "which URL to follow" and "what the surrounding graph structure looks like" in a single forward pass. This eliminates the blended scoring (DQN * 0.7 + UCB * 0.3) entirely -- the transformer learns the blend implicitly from the graph tokens.

## Hidden Gems (Papers)

- **Ghasemi & Crowley (2026) -- "Learning When to Trust"** is cited for bias reduction, but the buried insight is about _evaluator reliability_. Their CESA-LinUCB learns per-domain trust boundaries, meaning the scheduler can detect when a domain's reward signal is unreliable (e.g., the extraction pipeline fails on that domain's HTML structure, producing false negatives). This turns a bandit problem into a bandit-with-corrupted-feedback problem -- a much harder setting that UCB1 has no mechanism to handle. Applying their trust framework to the domain scheduler would prevent the crawler from permanently blacklisting domains that simply have unusual HTML.

- **Wang et al. (2025) -- "Beyond Message Passing: Neural Graph Pattern Machine"** claims +52% crawl efficiency, but the real contribution is the motif-aware GNN. Standard GNNs with message passing cannot count graph motifs (triangles, k-cliques) -- this is a proven expressivity limitation of the WL-isomorphism test. Their pattern machine can detect, for example, that a cluster of 5 company pages all link to the same "partners" directory -- a structure that screams "lead-rich hub" but is invisible to flat state vectors and even standard GNNs. This is the only paper in the cited set that addresses a fundamental expressivity limitation rather than just scaling an existing technique.

- **Kontogiannis et al. (2021) -- tree-based RL for crawling** is buried in the RESEARCH.md references but contains a key idea: modeling the crawl frontier as a _tree_ rather than a flat priority queue. Their tree-based Q-learning allows the agent to reason about entire subtrees of the web graph (e.g., "if I follow this directory page, I get access to 50 company profiles underneath it"). The current SQLite frontier has no concept of tree structure -- it treats every URL as independent. Grafting their tree-based value estimation onto the GNN state encoder would let the agent evaluate hub pages by their subtree potential, not just their immediate content.

- **Gavin & Zhang (2023) -- replay buffer pruning** seems routine, but their insight about _catastrophic forgetting from pruning_ is critical. When the replay buffer hits 100K and old transitions are deleted by timestamp, the agent can forget rare domain types it encountered early in training. Their pruning strategy preserves a "core set" of diverse high-reward transitions regardless of age. Without this, the current timestamp-based pruning will systematically delete the earliest (and often most diverse) exploration data.

## Contrarian Takes

- **The hierarchical PPO-DQN architecture (Liang 2025) is over-engineered for this problem.** PPO's advantage over DQN is in continuous action spaces and high-dimensional policy optimization. Domain selection from a discrete set of ~1000 domains is not a continuous action space -- it is a contextual bandit problem. NeuralUCB solves it with a fraction of the complexity. The GRU meta-controller for switching exploration/exploitation modes adds a third optimization loop with its own hyperparameters, training instability, and failure modes. The simpler architecture (DT for URL selection + NeuralUCB for domain selection) achieves comparable results with two components instead of three.

- **The "offline RL" advantage of Decision Transformers is overstated for this use case.** The research pitches DT's offline capability as a major advantage ("train on historical crawl logs without live interaction"). But the whole point of the crawler is that it _must_ interact live with the web. The real advantage of DT is not offline training -- it is that the sequence model captures multi-hop dependencies that Q-learning misses. Running the DT in online mode (updating on fresh trajectories as they arrive) is strictly better than offline-only training, and the existing LanceDB replay buffer already supports this. The "offline RL" framing is a distraction from the actual architectural advantage.

- **Curriculum learning (Priority 3 in the upgrade path) should come _before_ NeuralUCB, not after.** The upgrade path prioritizes NeuralUCB (Priority 2) over curriculum learning (Priority 3), but this is backwards. NeuralUCB needs ~100 observations per domain to produce calibrated uncertainty estimates. On hard domains (anti-bot, JS-heavy, low lead density), those 100 observations will be mostly failures, producing a biased reward predictor. Curriculum learning ensures the agent learns basic crawling competence on easy domains first, so when it encounters hard domains, the NeuralUCB reward predictor has a meaningful prior from transfer. Train easy first, then let NeuralUCB handle the hard-domain exploration.

- **The 448-dim state vector's biggest weakness is not dimensionality -- it is the embedding model.** All three documents treat `all-MiniLM-L6-v2` as a fixed choice. But this model was trained on general NLI/STS tasks, not on B2B lead page discrimination. Two pages that are semantically similar in general English ("Company X provides consulting services" vs "Company Y provides consulting services") will have nearly identical embeddings, even though one might be a lead and the other might not (different ICP fit). A contrastive fine-tune of the embedding model on (lead-page, non-lead-page) pairs from the crawler's own data would likely improve harvest rate more than any of the RL architecture changes.

## Wild Ideas

- **LLM-as-crawler-policy via tool-use.** Skip the DQN/DT entirely. Give a small local LLM (Qwen-2.5-3B, already deployed for email generation in the MLX pipeline) the page content, the list of outgoing links, and the instruction "select the link most likely to lead to a B2B decision-maker's contact page." The LLM reasons in natural language about link text, URL patterns, and page context. No state vector, no replay buffer, no training loop. The LLM's zero-shot performance on this task is likely competitive with a trained DQN's early-phase performance, and it can be distilled into a lightweight policy network once enough trajectories are collected. The DT then becomes the student, not the teacher.

- **World model for web prediction.** Train a small transformer to predict the next page's content features given the current page and the chosen link. This is not for curiosity reward -- it is for _look-ahead planning_. Before following a link, the agent can simulate 3-5 steps ahead using the world model and estimate the expected cumulative reward of the entire subtree. This turns greedy single-step URL selection into tree-search planning (like MCTS in AlphaGo, but for web graphs). The world model's prediction accuracy is surprisingly tractable because web graph structure is highly predictable: company directories lead to company pages, company pages link to team/about pages, team pages contain leads.

- **Diffusion models for frontier priority.** Instead of scoring each URL independently with a Q-value, treat the frontier as a _distribution_ over URL quality and use a denoising diffusion process to iteratively refine the priority ordering. Start with the uniform distribution over frontier URLs, then denoise through K steps using a learned score function conditioned on the current crawl state. The final "clean" distribution concentrates probability mass on the best URLs. This naturally handles correlations between URLs (e.g., two URLs from the same directory should not both be top-priority -- the diffusion process would de-correlate them). No existing crawling system does this.

- **Multi-agent competitive crawling.** Run N crawler agents on the same web graph, but each agent gets _negative reward_ when it visits a page another agent already claimed. This induces natural diversity pressure: agents automatically specialize into different domain niches, industry verticals, or geographic regions without any explicit assignment. The competitive pressure also solves the domain diversity collapse observed at high DQN weight (0.8/0.2 blend): agents that over-exploit a domain are punished by the other agents' prior visits. Implementation: a shared Bloom filter of visited URLs, with the penalty proportional to the number of agents that have already visited.

- **Active learning for the reward model.** The extraction pipeline is the bottleneck for reward signal. Instead of extracting leads from every crawled page, train a lightweight binary classifier ("does this page contain a lead?") and only send _uncertain_ pages to the full extraction pipeline. The classifier's uncertainty directly becomes the ICM curiosity signal: pages the classifier cannot confidently label are both the most informative for improving the classifier _and_ the most likely to contain novel lead types. This cuts extraction pipeline load by 80%+ while simultaneously improving the diversity of discovered leads.

- **Embedding-space reward shaping via optimal transport.** Define a "target distribution" in embedding space as the centroid cluster of all known lead pages. Shape the reward as the negative Wasserstein distance between the current page's embedding and the target distribution. As new leads are discovered, the target distribution updates, and the reward landscape shifts. This creates a continuously evolving gradient that pulls the crawler toward lead-rich regions of the web embedding space. Unlike fixed reward thresholds (+1.0, +0.2, -0.1), this provides a smooth, dense signal proportional to how "lead-like" a page's content is.

```python
class EpsilonScheduler:
    def __init__(self, start=1.0, end=0.01, decay_steps=100000):
        self.start = start
        self.end = end
        self.decay_steps = decay_steps
        self.decay_rate = (start - end) / decay_steps
    
    def get_epsilon(self, step):
        # Linear decay
        epsilon = max(self.end, self.start - step * self.decay_rate)
        # Or exponential decay: epsilon = self.end + (self.start - self.end) * exp(-step/self.decay_steps)
```

```python
class FrontierQueue:
    def __init__(self, max_size=1000000):
        self.priority_queue = []  # Min-heap for SQLite integration
        self.url_set = set()      # In-memory for fast lookups
        self.bloom_filter = BloomFilter(max_size, error_rate=0.001)
        self.domain_queues = {}   # Per-domain subqueues
        
    def add_url(self, url, priority, domain):
        """Add URL to frontier with deduplication"""
        url_hash = hashlib.md5(url.encode()).hexdigest()
        
        # Bloom filter check (fast, probabilistic)
        if self.bloom_filter.check(url_hash):
            # Possible false positive, check exact set
            if url in self.url_set:
                return False  # Duplicate
        
        # Add to all structures
        self.bloom_filter.add(url_hash)
        self.url_set.add(url)
        
        # Add to SQLite frontier table
        self._sqlite_add(url, domain, priority)
        
        # Maintain in-memory heap for quick access
        heapq.heappush(self.priority_queue, (-priority, url, domain))
        
        return True

        # Combine DQN Q-values with domain UCB scores
        candidates = []
        for _ in range(min(100, len(self.priority_queue))):
            priority, url, domain = heapq.heappop(self.priority_queue)
            domain_score = self.domain_scheduler.get_score(domain)
            combined_score = -priority * 0.7 + domain_score * 0.3

            for score, url, domain in candidates[1:]:
```

```python
import torch
import torch.nn as nn
import numpy as np

class NeuralUCB(nn.Module):
    def __init__(self, input_dim=512, hidden_dims=[256, 128, 64]):
        super().__init__()
        self.input_dim = input_dim
        
        # Mean network
        layers = []
        prev_dim = input_dim
        for hidden_dim in hidden_dims:
            layers.append(nn.Linear(prev_dim, hidden_dim))
            layers.append(nn.ReLU())
            prev_dim = hidden_dim
        layers.append(nn.Linear(prev_dim, 1))
        self.mean_network = nn.Sequential(*layers)
        
        # Uncertainty network
        self.uncertainty_network = nn.Sequential(
            nn.Linear(input_dim, 256),
            nn.ReLU(),
            nn.Linear(256, 128),
            nn.ReLU(),
            nn.Linear(128, 1),
            nn.Softplus()  # Ensure positive uncertainty
        )
        
        # Neural Tangent Kernel approximation
        self.ntk_scale = 0.1
        self.beta = 2.0  # Exploration parameter
        
    def forward(self, x, compute_uncertainty=True):
        mean = self.mean_network(x)
        
        if compute_uncertainty:
            uncertainty = self.uncertainty_network(x)
            # Neural Tangent Kernel-based uncertainty
            with torch.no_grad():
                jacobian = torch.autograd.functional.jacobian(
                    lambda x: self.mean_network(x).sum(), x
                )
                ntk_uncertainty = torch.norm(jacobian, dim=1) * self.ntk_scale
                total_uncertainty = uncertainty + ntk_uncertainty.unsqueeze(-1)
            
            ucb_score = mean + self.beta * total_uncertainty
            return ucb_score, mean, total_uncertainty
        else:
            return mean

            optimizer.zero_grad()
            loss.backward()
            optimizer.step()
    
    def select_domain(self, candidate_domains):
        """Select domain using NeuralUCB scores"""
        scores = []
        for domain in candidate_domains:
            stats = get_domain_stats(domain)
            features = self.extract_domain_features(domain, stats)
            with torch.no_grad():
                ucb_score, _, _ = self.neural_ucb(features.unsqueeze(0))
                scores.append(ucb_score.item())
        
        # Softmax selection for exploration
        probs = torch.softmax(torch.tensor(scores) * 2.0, dim=0)
        selected_idx = torch.multinomial(probs, 1).item()
        
        return candidate_domains[selected_idx]
```

### 3.1 Architecture

class IntrinsicCuriosityModule(nn.Module):
    def __init__(self, state_dim=448, action_dim=10, hidden_dim=256):
        super().__init__()
        self.state_dim = state_dim
        self.action_dim = action_dim
        
        # Feature encoder φ(s)
        self.feature_encoder = nn.Sequential(
            nn.Linear(state_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim // 2),
            nn.ReLU(),
            nn.Linear(hidden_dim // 2, 128)  # Compact feature space
        )
        
        # Inverse model: (φ(s_t), φ(s_{t+1})) → a_t
        self.inverse_model = nn.Sequential(
            nn.Linear(256, 128),  # 128*2 = 256
            nn.ReLU(),
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, action_dim)
        )
        
        # Forward model: (φ(s_t), a_t) → φ(s_{t+1})
        self.forward_model = nn.Sequential(
            nn.Linear(128 + action_dim, 128),  # φ(s_t) + one-hot action
            nn.ReLU(),
            nn.Linear(128, 128),
            nn.ReLU(),
            nn.Linear(128, 128)  # Predict φ(s_{t+1})
        )
        
        # Curiosity scaling parameter
        self.curiosity_scale = 0.1
        
    def encode_state(self, state):
        return self.feature_encoder(state)
    
    def compute_curiosity(self, state, action, next_state):
        # Encode states
        phi_t = self.encode_state(state)
        phi_t1 = self.encode_state(next_state)
        
        # Forward model prediction
        action_onehot = F.one_hot(action, self.action_dim).float()
        forward_input = torch.cat([phi_t, action_onehot], dim=-1)
        predicted_phi_t1 = self.forward_model(forward_input)
        
        # Curiosity reward = prediction error
        curiosity_reward = F.mse_loss(predicted_phi_t1, phi_t1, reduction='none')
        curiosity_reward = curiosity_reward.mean(dim=-1)  # Average over features
        
        return curiosity_reward * self.curiosity_scale
    
    def compute_losses(self, state, action, next_state):
        phi_t = self.encode_state(state)
        phi_t1 = self.encode_state(next_state)
        
        # Inverse model loss
        inverse_input = torch.cat([phi_t, phi_t1], dim=-1)
        predicted_action = self.inverse_model(inverse_input)
        inverse_loss = F.cross_entropy(predicted_action, action)
        
        # Forward model loss
        action_onehot = F.one_hot(action, self.action_dim).float()
        forward_input = torch.cat([phi_t, action_onehot], dim=-1)
        predicted_phi_t1 = self.forward_model(forward_input)
        forward_loss = F.mse_loss(predicted_phi_t1, phi_t1)
        
        return inverse_loss, forward_loss

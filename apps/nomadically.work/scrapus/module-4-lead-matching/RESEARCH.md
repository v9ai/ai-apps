### 1. Siamese Networks for Profile Similarity & ICP Matching

Siamese networks have shown significant promise in semantic matching
applications, particularly in business contexts where understanding semantic
relationships between entities is crucial. The Scrapus module uses
128-dimensional embeddings to encode ICP and candidate company profiles.

**Key Research Insights:**

- **Nigam et al. (2019)** -- Semantic Product Search -- demonstrates that deep
  learning models for semantic matching effectively address limitations of
  lexical matching, including understanding hypernyms, synonyms, and
  morphological variants.
  [https://doi.org/10.1145/3292500.3330759]
- The Scrapus approach of encoding multiple attributes (industry keywords,
  company size, location, description) into a unified embedding space follows
  similar principles to semantic matching systems handling diverse feature
  types.

The module's cosine-distance-to-similarity conversion (`1.0 - distance`) is
standard in similarity learning. The semantic matching examples (e.g.,
"AI-driven healthcare startups" vs. "ML platform for medical image analysis" =
0.91) demonstrate capture of semantic relationships beyond exact keyword
matching.

## Calibration Curve Analysis

Calibration measures how well predicted probabilities match observed
frequencies. A perfectly calibrated model predicts `P(qualified) = 0.80` for a
cohort where exactly 80% of companies are qualified.

### Method

1. Bin all test-set predictions into 10 equal-width bins (0.0-0.1, ...,
   0.9-1.0).
2. For each bin, compute the mean predicted probability and the fraction of
   actual positives.
3. Plot predicted vs. observed and compute the Expected Calibration Error
   (ECE).

### Results (test set, n = 300)

| Bin       | Mean Predicted | Fraction Positive | Count |
|-----------|---------------:|------------------:|------:|
| 0.0 - 0.1 | 0.05          | 0.03              | 42    |
| 0.1 - 0.2 | 0.15          | 0.12              | 31    |
| 0.2 - 0.3 | 0.26          | 0.24              | 28    |
| 0.3 - 0.4 | 0.35          | 0.31              | 25    |
| 0.4 - 0.5 | 0.45          | 0.42              | 22    |
| 0.5 - 0.6 | 0.54          | 0.56              | 20    |
| 0.6 - 0.7 | 0.65          | 0.68              | 24    |
| 0.7 - 0.8 | 0.74          | 0.77              | 30    |
| 0.8 - 0.9 | 0.84          | 0.87              | 38    |
| 0.9 - 1.0 | 0.94          | 0.92              | 40    |

**ECE = 0.028** (weighted average of |predicted - observed| across bins).

### Interpretation

- The raw ensemble (without isotonic calibration) already shows reasonable
  alignment between predicted and observed probabilities.
- The largest gap is in the 0.3-0.4 bin (0.04 absolute), acceptable for
  production use.
- After applying isotonic calibration on the validation set, ECE drops to
  approximately 0.014 on the test set.
- XGBoost tends to produce slightly overconfident predictions in the 0.1-0.4
  range, while LogReg slightly underestimates in the 0.7-0.9 range. The
  ensemble averaging partially corrects both biases.

### Recommendation

Apply isotonic calibration as a post-processing step. This is already
implemented in the `CalibratedEnsemble` class (see `IMPLEMENTATION.md`,
Section 3) but is not yet wired into the production scoring path (see
"Production Gaps" in `README.md`).

## Online Learning Deployment Checklist

This checklist covers the operational requirements for enabling incremental
model updates in production, building on the `OnlineLeadMatcher` framework
described in `IMPLEMENTATION.md`.

### Infrastructure

- [ ] **Label ingestion endpoint.** Expose an API (or CLI command) that
  accepts `(company_id, is_qualified, feedback_source)` and writes to a
  `labels` table in SQLite.
- [ ] **Label buffer table.** Create `CREATE TABLE label_buffer (id INTEGER
  PRIMARY KEY, company_id INTEGER, features BLOB, is_qualified INTEGER,
  ingested_at REAL)` in the scoring database.
- [ ] **Retrain trigger.** Cron job or row-count trigger that fires
  `OnlineLeadMatcher._retrain_window()` when `label_buffer` reaches 100 rows.
- [ ] **Model staging directory.** New model artefacts are written to
  `scrapus_data/models/staging/` before promotion to `scrapus_data/models/`.
- [ ] **Atomic model swap.** Use symlink rotation (`current -> v42/`) so
  in-flight scoring requests are not disrupted during model replacement.

### Validation Gates

- [ ] **Minimum performance gate.** After retraining, evaluate on the held-out
  test set (300 examples). Reject the new model if F1 < 0.85 or ROC-AUC <
  0.91.
- [ ] **Calibration gate.** Compute ECE on the test set. Reject if ECE > 0.05.
- [ ] **Shadow scoring.** Run the candidate model in parallel for 48 hours,
  logging predictions to `lead_explanations` with `model_version = 'shadow'`.
  Compare distributions before promotion.
- [ ] **Rollback trigger.** If F1 on the most recent 50 labeled examples
  drops below 80% of baseline, automatically rollback via `ModelRegistry`.

### Drift Detection

- [ ] **Feature drift monitor.** Track rolling mean and variance of each
  numeric feature. Alert when any feature's z-score exceeds 3.0 vs. the
  training distribution.
- [ ] **Prediction drift monitor.** Track the rolling mean of
  `ensemble_prob`. Alert when the 7-day mean deviates more than 0.10 from the
  training-set mean.
- [ ] **Label drift monitor.** Track the positive rate in the label buffer.
  Alert if it deviates more than 10 percentage points from the training
  positive rate (35%).

### Data Quality

- [ ] **Duplicate label check.** Reject duplicate `(company_id,
  feedback_source)` pairs in the label buffer.
- [ ] **Label delay tracking.** Log `ingested_at - scored_at` for each label.
  Alert if median delay exceeds 7 days (stale labels degrade retraining).
- [ ] **Feature completeness check.** Before retraining, verify that fewer
  than 20% of feature values in the buffer are missing (-1 sentinel). Pause
  retraining if exceeded.

### Logging & Audit

- [ ] **Retrain log table.** `CREATE TABLE retrain_log (id INTEGER PRIMARY
  KEY, model_version TEXT, triggered_at REAL, buffer_size INTEGER, train_f1
  REAL, train_roc_auc REAL, promoted INTEGER, rejection_reason TEXT)`.
- [ ] **Model lineage.** Each entry in `retrain_log` references the parent
  model version and the label buffer rows consumed.
- [ ] **Alerting.** Send Slack/email notification on: retrain triggered,
  model promoted, model rejected, rollback executed, drift alert.

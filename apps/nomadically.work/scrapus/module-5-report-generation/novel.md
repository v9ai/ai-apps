# Novel Insights -- Module 5: Report Generation

## Surprising Findings

- **Hallucination is mathematically inevitable** (Xu et al., 2024). No amount of prompt engineering or retrieval can guarantee zero hallucination. The Scrapus pipeline's pragmatic approach -- grounding + post-hoc verification -- is provably the correct strategy class. Chasing "zero hallucination" is wasted engineering effort; the only tractable goal is detection and mitigation with known error bounds.

- **More retrieval can actively degrade report quality.** The pipeline's conditional enrichment (ChromaDB context only when SQLite facts < 3) is backed by Self-RAG research: Asai et al. (2023) showed that indiscriminate retrieval introduces noise that increases hallucination rates. The counterintuitive finding is that for companies with rich structured data, skipping vector retrieval entirely produces more faithful reports than always retrieving top-k=5.

- **Token overlap at 0.5 threshold is a surprisingly effective hallucination detector despite being trivial.** The pipeline's regex-based claim extraction + token overlap verification achieves 97% factual accuracy (212/219 claims). NLI-based entailment (DeBERTa) would add ~15 seconds per report for diminishing returns on already-grounded short-form text. The lesson: for constrained outputs under ~60 words with strong source grounding, simple lexical methods dominate because the generation space is too small for sophisticated semantic drift.

- **Temperature 0.3 is not optimal for factual generation -- it's a compromise.** Temperature 0 produces degenerate repetitions in structured JSON output (repeat_penalty 1.1 partially masks this). Temperature 0.3 + top_p 0.9 is not a "factual setting" but a hack around the JSON formatting failure mode. Constrained decoding (Outlines/XGrammar) eliminates this tradeoff entirely: you can use temperature 0 for factual content because the grammar constraint prevents degenerate tokens.

- **GraphRAG's biggest win is not accuracy but hallucination reduction.** Han et al. (2025) showed GraphRAG drops hallucination from 12% to 4% while only improving accuracy by ~8%. The graph structure acts as an implicit constraint on what the LLM can plausibly say -- it cannot fabricate relationships that don't exist in the graph. This is a fundamentally different mechanism from retrieval accuracy improvements.

## Unconventional Techniques

- **Report generation via multi-agent debate.** Ghafarollahi & Buehler (2024) showed 63% improvement on complex analytical tasks by having specialist agents (financial analyst, team evaluator, market analyst) independently analyze the same data, then having a synthesis agent merge their outputs while a verification agent challenges unsupported claims. The key insight: the debate structure catches errors that any single agent would miss because each specialist has a different prior over what matters. This is not ensemble averaging -- it is adversarial collaboration.

- **Narrative graph traversal for report structure.** Instead of "retrieve chunks, stuff into prompt, generate," traverse the company knowledge graph to discover the narrative arc: `funding_event -> hiring_surge -> product_launch -> partnership`. The report structure emerges from the graph topology, not from a template. Multi-hop paths (up to 3 hops) surface relationships invisible to flat retrieval: "CEO previously at competitor X which raised Series C" becomes a competitive intelligence signal that no vector similarity search would find.

- **Report as constrained decoding, not generation + validation.** The current 5-stage validation pipeline with 2 retries is the wrong abstraction. XGrammar (Dong et al., 2024) and Outlines enforce JSON schema at the token level via context-free grammar constraints. The LLM literally cannot emit an invalid token. This converts a probabilistic pipeline (generate, validate, retry) into a deterministic one (generate valid output on the first pass). Retries become architecturally impossible to need.

- **RAPTOR's recursive summarization creates retrieval abstractions the LLM can reason over.** Instead of retrieving raw chunks, RAPTOR clusters and summarizes them into a tree. You retrieve from the appropriate abstraction level: leaf nodes for specific facts ("raised $10M"), mid-level for patterns ("rapid hiring across engineering"), root for strategic positioning ("high-growth AI security startup"). The query itself determines the retrieval depth. This is fundamentally different from reranking -- it changes what is retrievable, not just what is ranked highest.

- **Self-RAG reflection tokens as inline circuit breakers.** Rather than post-hoc hallucination checking (extract claims, verify, flag), the LLM emits `[RETRIEVE]`, `[CRITIQUE]`, `[SUPPORT]`, `[CONTRADICT]` tokens during generation. When it hits a `[CONTRADICT]` token, it stops, retrieves additional evidence, and regenerates that specific claim. The hallucination check happens mid-generation, not after. This turns a pipeline into a feedback loop -- architecturally superior because it prevents hallucinated text from influencing subsequent tokens via autoregressive conditioning.

## Hidden Gems (Papers)

- **Liu et al. (2025) -- E-Verify: Embedding-based factuality verification that scales to thousands of claims per second.** Everyone knows about NLI-based entailment checking (DeBERTa). E-Verify sidesteps the O(claims x facts) inference bottleneck by doing a single embedding pass and using vector similarity. At scale (100+ companies, 10+ claims each), this is 100x faster than pairwise NLI and nearly as accurate.

- **Saxena et al. (2024) -- Attribution in Scientific Literature: benchmarks for source attribution in LLM outputs.** Provides evaluation methodology for measuring whether generated text correctly attributes claims to specific sources. Directly applicable to Scrapus's `sources` field in the JSON schema, which is currently just URL pass-through with no verification that the cited URL actually supports the generated claim.

- **HopRAG (2025) -- Multi-hop reasoning with retrieve-reason-prune.** Goes beyond basic graph traversal. At each hop, the model reasons about whether to continue traversing or prune the path. This prevents the combinatorial explosion problem in 3-hop queries where naive graph traversal returns hundreds of irrelevant nodes. The pruning is LLM-guided, not heuristic.

- **Takehi et al. (2025) -- mxbai-edge-colbert: ColBERT that runs on CPU.** ColBERT's late-interaction scoring (per-token MaxSim) was previously GPU-only in practice. This paper shows a quantized variant with minimal quality loss that makes ColBERT viable for Ollama-class local deployments. Replaces the single-vector cosine similarity in ChromaDB with token-level matching that captures fine-grained semantic relationships without a GPU.

- **Wu et al. (2025) -- SourceCheckup: citation error rate dropped from 50% to 12%.** The surprising number is the baseline: 50% of LLM citations are wrong even when sources are provided. This means Scrapus's `sources` field in generated reports is likely wrong half the time. SourceCheckup introduces an agent-based pipeline that checks whether the cited source actually contains evidence for the claim, not just whether the claim appears grounded.

## Contrarian Takes

- **The cosine threshold of 0.3 is almost certainly too low, and raising it would improve reports.** With 384-dim MiniLM embeddings, 0.3 cosine similarity represents a barely-above-random relationship. The pipeline injects this noise as "background context" to the LLM. Raising the threshold to 0.5 or 0.6 would discard more documents but the remaining ones would actually be relevant. The conditional enrichment logic (only use ChromaDB when facts < 3) implicitly acknowledges this problem but doesn't fix it for the case where ChromaDB is queried.

- **Two few-shot exemplars in the system prompt may be hurting more than helping for local models.** With an 8192-token context window and a 6000-token prompt budget, two full JSON exemplars consume ~400 tokens of precious context. For GPT-4 with 128K context this is negligible; for llama3.1:8b with 8K context this is 5% of the window and directly competes with source facts. The structured decoding approach (Outlines) eliminates the need for exemplars entirely because the schema is enforced at the token level.

- **The retry-on-failure validation pattern trains the LLM on its own errors.** When the pipeline appends "Please output valid JSON" and re-prompts after a parse failure, the second prompt includes the instruction to fix an error the model made. But the model doesn't know what error it made -- it just sees a longer prompt. The retry is not corrective, it's just a re-roll with a slightly different prompt. Constrained decoding eliminates retries by construction, but more importantly, it eliminates the illusion that retries are a correction mechanism.

- **Confidence scores generated by the LLM are meaningless without calibration.** The pipeline asks the LLM to output a `confidence` float in [0, 1], but LLMs have no calibrated sense of their own uncertainty. The README acknowledges this gap (Platt scaling/isotonic regression needed) but the deeper issue is that LLM-generated confidence and actual factual accuracy are nearly uncorrelated for short-form generation. A heuristic based on source fact count and token overlap scores would be more reliable than the LLM's self-assessment.

- **Cross-encoder reranking is overkill for this pipeline.** The planned upgrade to ms-marco-MiniLM adds 500ms per report for 10 candidates. But the pipeline only retrieves 5 documents, filters by cosine > 0.3, then reranks by fact overlap. After filtering, there are typically 2-3 documents left. Reranking 2-3 documents with a cross-encoder is wasted compute -- the token overlap reranker is sufficient at this scale. The real retrieval bottleneck is that ChromaDB uses single-vector similarity, not the reranking step.

## Wild Ideas

- **Reports that update themselves via streaming RAG.** Instead of static batch-generated reports, emit a report as a live document that subscribes to the company's ChromaDB collection. When new pages are crawled for that company, the report automatically re-runs claim verification against the new data. Claims that are now contradicted get flagged. Claims that are newly supported get their confidence boosted. The report is never "generated" -- it is continuously maintained. The Self-RAG reflection loop runs in the background on new evidence.

- **Adversarial report testing via red-team LLM.** After generating a report, pass it to a second LLM whose job is to find the most plausible-sounding false claim. This adversary has access to the same source data and tries to determine which claims the generator fabricated vs. grounded. Claims that fool the adversary (appear grounded but aren't) are the dangerous ones. This is a minimax game: the generator learns to produce only verifiable claims because the adversary will catch anything else.

- **Report quality as RL reward signal for the local model.** Use the citation verification score (verification_rate from the embedding-based pipeline) as a reward signal. Fine-tune the local llama3.1:8b with DPO (Direct Preference Optimization): generate two reports for the same company, use the one with higher verification_rate as the preferred completion. Over 100-200 companies, the model learns to generate claims that are more likely to be grounded. This closes the loop between verification and generation -- the hallucination detector becomes the training signal.

- **Fact-budget prompting.** Instead of truncating the prompt when it exceeds 6000 tokens, give the LLM a "fact budget": "You may make exactly N factual claims in this report, where N = number of source facts." If the company has 4 facts, the report gets 4 claims. If it has 12 facts, the report gets 12 claims and is longer. The report length adapts to evidence density rather than being a fixed 60 words. This makes the confidence score meaningful: a 4-claim report from 4 facts should be high-confidence, while a 4-claim report from 12 facts should indicate the LLM selectively summarized.

- **Knowledge graph diffing for competitive intelligence.** Build knowledge graphs for all companies in the pipeline. Diff two company graphs to find structural gaps: "Company A has a VP of AI but Company B doesn't," "Company A has 3 funding events but Company B has 1." These structural differences become report content automatically -- no LLM needed for the analysis step. The LLM only renders the diff into natural language. This inverts the typical RAG pattern: instead of "retrieve text, generate insight," it's "compute insight structurally, generate text."
